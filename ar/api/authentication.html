المصادقة

يصادَق كل طلب إلى واجهة Temporis البرمجية برمز حامل (bearer token). إليك كيفية إنشائه وإرساله والحفاظ على أمانه.

رموز الوصول

تنشئ رموز الوصول في لوحة التحكم، ضمن رموز الوصول. الرمز عبارة عن سلسلة عشوائية طويلة مسبوقة بـ ts_.

يُعرَض الرمز الكامل مرة واحدة، في لحظة إنشائه. لا يخزّن Temporis سوى تجزئة (hash) منه على الخادم، ولا يخزّن الرمز نفسه أبدًا. وهذا يعني أن الرمز المفقود لا يمكن استرجاعه — إذا فقدته، فاحذفه وأنشئ رمزًا جديدًا.

عامل الرموز معاملة كلمات المرور

يستطيع أي شخص يملك أحد رموزك أن يدخل البيانات ويشغّل التنبؤات — والتنبؤات تُحتسب على حسابك. فاحرص على حمايتها وفقًا لذلك.

مصادقة طلب

أرسل الرمز في ترويسة Authorization مع كل طلب، باستخدام نظام Bearer:

Authorization: Bearer ts_your_token_here

تقرأ الأمثلة في كامل هذا التوثيق الرمز من متغير بيئة TEMPORIS_TOKEN، حتى لا يظهر السر أبدًا في الشيفرة المصدرية. صدّره مرة واحدة في صدفتك (shell):

export TEMPORIS_TOKEN="ts_your_token_here"

ثم أرفقه بوصفه ترويسة:

curl https://api.temporis.co/v1/predict \
  -H "Authorization: Bearer $TEMPORIS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{ "data_profile": "hourly_orders", "count": 5, "temperature": 1.0, "top_p": 0.9 }'
import os, requests

token = os.environ["TEMPORIS_TOKEN"]

resp = requests.post(
    "https://api.temporis.co/v1/predict",
    headers={"Authorization": f"Bearer {token}"},
    json={"data_profile": "hourly_orders", "count": 5, "temperature": 1.0, "top_p": 0.9},
)
resp.raise_for_status()
const token = process.env.TEMPORIS_TOKEN;

const resp = await fetch("https://api.temporis.co/v1/predict", {
  method: "POST",
  headers: {
    "Authorization": `Bearer ${token}`,
    "Content-Type": "application/json",
  },
  body: JSON.stringify({ data_profile: "hourly_orders", count: 5, temperature: 1.0, top_p: 0.9 }),
});

عند فشل المصادقة

إذا كانت ترويسة Authorization مفقودة أو غير صحيحة التنسيق، أو كان الرمز غير معروف، تستجيب الواجهة البرمجية بالرمز HTTP 401 وجسم JSON يصف الخطأ الذي وقع:

{ "detail": "Invalid access token." }

أما الترويسة المفقودة أو غير صحيحة التنسيق فتُرجع { "detail": "Missing access token." } بدلًا من ذلك. وكلاهما من النوع 401. راجع الأخطاء ورموز الحالة للاطلاع على القائمة الكاملة للاستجابات.

الحفاظ على أمان الرموز

بعض العادات تبقي رموزك بعيدة عن الأيدي الخاطئة:

  • لا تودِع الرموز أبدًا في نظام إدارة الشيفرة المصدرية، ولا تضعها أبدًا في الشيفرة من جهة العميل. فالمتصفحات وتطبيقات الجوال تشحن شيفرتها المصدرية إلى المستخدمين.
  • اقرأ الرموز من متغيرات البيئة أو من مدير أسرار. أبقها بعيدة عن السجلات ولقطات الشاشة والتذاكر.
  • أنشئ رمزًا منفصلًا لكل خدمة حتى تتمكن من إبطال واحد دون تعطيل الباقي.
  • احذف رمزًا في لوحة التحكم لإبطاله فورًا. يسري الحذف على الفور — وأول طلب يستخدم ذلك الرمز يحصل على 401.
