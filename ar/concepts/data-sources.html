مصادر البيانات

مصدر البيانات هو المكان الذي تعيش فيه أرصادك الخام. تغطي هذه الصفحة نموذج بيانات Temporis وكيفية إرسال البيانات إليه على نحو جيّد.

تنشئ مصدر بيانات مرة واحدة في لوحة التحكم، ثم تبثّ الأرصاد إليه عبر الواجهة البرمجية مع حدوثها.

نموذج البيانات

يخزّن Temporis البيانات في صورة صفوف. وكل رصد ثلاثية واحدة: متى قِيس شيء ما، وما الذي قِيس، والرقم الذي قِسته.

timestamp
وقت يونكس بالـثواني (عدد صحيح). اللحظة التي أُخذ فيها الرصد.
name
السلسلة التي تنتمي إليها هذه القيمة — مثل orders أو revenue.
value
رقم. والأعداد العشرية مقبولة.

يستطيع مصدر بيانات واحد أن يحتفظ بسلاسل مسمّاة كثيرة جنباً إلى جنب. والحقل name هو ما يميّزها. وبياناتك معزولة عن بيانات كل عميل آخر — فمصدر البيانات يخصّك وحدك.

إنشاء مصدر بيانات

تنشئ مصدر بيانات في لوحة التحكم تحت مصادر البيانات. امنحه اسماً ويصبح جاهزاً لاستقبال البيانات.

تتبع الأسماء بضع قواعد:

  • تبدأ بحرف صغير.
  • استخدم الحروف الصغيرة والأرقام والشرطات السفلية فقط.
  • حتى 48 محرفاً.

فـstore_metrics وgrid_load_eu صالحان؛ أما Store-Metrics و2024data فليسا كذلك. ويبقى مصدر البيانات المُنشأ حديثاً فارغاً حتى ترسل إليه أرصاداً.

إرسال الأرصاد

تضيف البيانات بنقطة نهاية الاستيعاب. ويحمل كل نداء مصدر البيانات الهدف data_source وقائمة من records. هنا ندفع بضع قراءات orders إلى store_metrics.

curl -X POST https://api.temporis.co/v1/data_sources/ingest \
  -H "Authorization: Bearer ts_<token>" \
  -H "Content-Type: application/json" \
  -d '{
    "data_source": "store_metrics",
    "records": [
      { "timestamp": 1718870400, "name": "orders", "value": 42 },
      { "timestamp": 1718874000, "name": "orders", "value": 51 },
      { "timestamp": 1718877600, "name": "orders", "value": 47 }
    ]
  }'
import requests

requests.post(
    "https://api.temporis.co/v1/data_sources/ingest",
    headers={"Authorization": "Bearer ts_<token>"},
    json={
        "data_source": "store_metrics",
        "records": [
            {"timestamp": 1718870400, "name": "orders", "value": 42},
            {"timestamp": 1718874000, "name": "orders", "value": 51},
            {"timestamp": 1718877600, "name": "orders", "value": 47},
        ],
    },
)  # -> 204 No Content
await fetch("https://api.temporis.co/v1/data_sources/ingest", {
  method: "POST",
  headers: {
    "Authorization": "Bearer ts_<token>",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    data_source: "store_metrics",
    records: [
      { timestamp: 1718870400, name: "orders", value: 42 },
      { timestamp: 1718874000, name: "orders", value: 51 },
      { timestamp: 1718877600, name: "orders", value: 47 },
    ],
  }),
}); // -> 204 No Content

يُعيد الاستيعاب الناجح HTTP 204 No Content بجسم فارغ — فليس هناك ما يُقرأ مجدداً، إذ تُخزَّن الصفوف ببساطة. راجع مرجع الاستيعاب للاطلاع على تفاصيل الطلب والأخطاء كاملةً.

سلاسل كثيرة في مصدر واحد

لست مقيّداً بسلسلة واحدة لكل مصدر. أرسل قيم name مختلفة إلى data_source نفسه وستتعايش كسلاسل منفصلة. فقد يتتبّع متجر الطلبات والإيرادات والزائرين جميعاً في store_metrics.

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718870400, "name": "orders",  "value": 42 },
    { "timestamp": 1718870400, "name": "revenue", "value": 1284.50 }
  ]
}

إبقاء السلاسل المترابطة معاً مريح ويتيح لك التنبؤ بها مجتمعةً لاحقاً. وملف البيانات هو المكان الذي تختار فيه أيّاً من هذه الأسماء ستقرؤه فعلاً.

التحديثات والتعبئة الرجعية

مفتاح الإدراج-أو-التحديث هو الزوج (timestamp, name). وإرسال سجل بطابع زمني واسم موجودين سلفاً يستبدل القيمة المخزّنة بدلاً من إضافة نسخة مكرّرة.

التعبئة الرجعية وإعادة المحاولة آمنتان

لأن عمليات الكتابة عديمة الأثر التراكمي على (timestamp, name)، يمكنك إعادة تشغيل دفعة بعد تعثّر في الشبكة، أو تصحيح قراءة خاطئة، أو تحميل أشهر من التاريخ بأي ترتيب — والنتيجة دائماً القيمة النهائية نفسها لكل سلّة.

الأرشفة

عندما يتوقف مصدر عن جمع بيانات جديدة، يمكنك أرشفته في لوحة التحكم. توقف الأرشفة الاستيعاب اللاحق: فطلب يستهدف مصدراً مؤرشفاً يُعيد HTTP 409 "Data source is archived."

الأرشفة غير مُتلِفة. فالبيانات الموجودة تبقى في مكانها، وأي ملفات بيانات بُنيت على المصدر تستمر في العمل مقابل البيانات المخزّنة سلفاً.
