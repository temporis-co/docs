Fehler & Statuscodes

Wie Temporis Erfolg und Fehler meldet, was jeder Status bedeutet und wie Sie sie behandeln.

Form der Antwort

Die Form einer Antwort hängt davon ab, was passiert ist:

  • Ein erfolgreicher Ingest gibt 204 No Content ohne Body zurück.
  • Eine erfolgreiche Prognose gibt 200 mit einem JSON-Body zurück.
  • Jeder Fehler gibt den relevanten HTTP-Status mit einem JSON-Body der Form { "detail": "..." } zurück.
{ "detail": "Data profile not found." }

Fehler behandeln

Die meisten Fehler lassen sich einer klaren nächsten Aktion zuordnen:

  • 401 — prüfen Sie, ob der Authorization-Header vorhanden und der Token gültig ist. Siehe Authentifizierung.
  • 402 — fügen Sie im Dashboard unter Billing eine Kreditkarte hinzu oder korrigieren Sie sie und versuchen Sie es erneut.
  • 404 — stellen Sie sicher, dass die Datenquelle oder das Datenprofil existiert und dass das Profil bereit zur Auslieferung ist.
  • 409 bei predict — nehmen Sie mehr Historie auf, bis das Profil sein Minimum erreicht, und versuchen Sie es erneut.
  • 409 bei ingest — die Datenquelle ist archiviert; heben Sie die Archivierung auf oder schreiben Sie in eine andere Quelle.
  • 502 — vorübergehend. Versuchen Sie es mit exponentiellem Backoff erneut.

Eine einfache Verzweigung nach dem Statuscode deckt all diese Fälle ab:

import time, requests

def predict(payload, token, retries=3):
    for attempt in range(retries):
        resp = requests.post(
            "https://api.temporis.co/v1/predict",
            headers={"Authorization": f"Bearer {token}"},
            json=payload,
        )
        if resp.status_code == 200:
            return resp.json()
        if resp.status_code == 502:
            time.sleep(2 ** attempt)   # transient: back off and retry
            continue
        detail = resp.json().get("detail", "")
        if resp.status_code == 402:
            raise RuntimeError(f"Billing: {detail} — add a card in the dashboard.")
        if resp.status_code == 409:
            raise RuntimeError(f"Not ready: {detail} — ingest more history.")
        resp.raise_for_status()        # 400 / 401 / 404: fix the request
    raise RuntimeError("Prediction failed after retries.")

Wiederholungen

Das Richtige wiederholen

Ein 502 kann gefahrlos wiederholt werden — er ist vorübergehend. Ingest ist idempotent (Schreibvorgänge führen ein Upsert auf dem Schlüssel (timestamp, name) durch), daher ist das Wiederholen eines fehlgeschlagenen Ingest immer sicher. Vermeiden Sie es, 4xx-Antworten blind zu wiederholen; korrigieren Sie zuerst die Anfrage.
