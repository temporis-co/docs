Datenquellen

Eine Datenquelle ist der Ort, an dem Ihre Roh-Beobachtungen liegen. Diese Seite behandelt das Datenmodell von Temporis und wie Sie Daten richtig hineinsenden.

Sie erstellen eine Datenquelle einmal im Dashboard und streamen dann Beobachtungen über die API hinein, sobald sie anfallen.

Das Datenmodell

Temporis speichert Daten als Zeilen. Jede Beobachtung ist ein einzelnes Tripel: wann etwas gemessen wurde, was gemessen wurde und die Zahl, die Sie gemessen haben.

timestamp
Unix-Zeit in Sekunden (eine Ganzzahl). Der Zeitpunkt, zu dem die Beobachtung erfasst wurde.
name
Die Reihe, zu der dieser Wert gehört — zum Beispiel orders oder revenue.
value
Eine Zahl. Fließkommawerte sind in Ordnung.

Eine einzelne Datenquelle kann viele benannte Reihen nebeneinander aufnehmen. Das Feld name ist es, das sie auseinanderhält. Ihre Daten sind von denen jedes anderen Kunden isoliert — eine Datenquelle gehört allein Ihnen.

Eine Datenquelle erstellen

Sie erstellen eine Datenquelle im Dashboard unter Data Sources. Geben Sie ihr einen Namen, und sie ist bereit, Daten zu empfangen.

Namen folgen einigen Regeln:

  • Beginnen Sie mit einem Kleinbuchstaben.
  • Verwenden Sie nur Kleinbuchstaben, Ziffern und Unterstriche.
  • Bis zu 48 Zeichen.

So sind store_metrics und grid_load_eu gültig; Store-Metrics und 2024data sind es nicht. Eine frisch erstellte Datenquelle ist leer, bis Sie ihr Beobachtungen senden.

Beobachtungen senden

Sie fügen Daten über den Ingest-Endpoint hinzu. Jeder Aufruf trägt die Ziel-data_source und eine Liste von records. Hier schieben wir ein paar orders-Messwerte in store_metrics.

curl -X POST https://api.temporis.co/v1/data_sources/ingest \
  -H "Authorization: Bearer ts_<token>" \
  -H "Content-Type: application/json" \
  -d '{
    "data_source": "store_metrics",
    "records": [
      { "timestamp": 1718870400, "name": "orders", "value": 42 },
      { "timestamp": 1718874000, "name": "orders", "value": 51 },
      { "timestamp": 1718877600, "name": "orders", "value": 47 }
    ]
  }'
import requests

requests.post(
    "https://api.temporis.co/v1/data_sources/ingest",
    headers={"Authorization": "Bearer ts_<token>"},
    json={
        "data_source": "store_metrics",
        "records": [
            {"timestamp": 1718870400, "name": "orders", "value": 42},
            {"timestamp": 1718874000, "name": "orders", "value": 51},
            {"timestamp": 1718877600, "name": "orders", "value": 47},
        ],
    },
)  # -> 204 No Content
await fetch("https://api.temporis.co/v1/data_sources/ingest", {
  method: "POST",
  headers: {
    "Authorization": "Bearer ts_<token>",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    data_source: "store_metrics",
    records: [
      { timestamp: 1718870400, name: "orders", value: 42 },
      { timestamp: 1718874000, name: "orders", value: 51 },
      { timestamp: 1718877600, name: "orders", value: 47 },
    ],
  }),
}); // -> 204 No Content

Ein erfolgreicher Ingest liefert HTTP 204 No Content mit leerem Body — es gibt nichts zurückzulesen, die Zeilen werden einfach gespeichert. Siehe die Ingest-Referenz für die vollständige Anfrage und Fehlerdetails.

Viele Reihen in einer Quelle

Sie sind nicht auf eine Reihe pro Quelle beschränkt. Senden Sie verschiedene name-Werte in dieselbe data_source, und sie koexistieren als separate Reihen. Ein Geschäft könnte Orders, Revenue und Visitors allesamt in store_metrics erfassen.

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718870400, "name": "orders",  "value": 42 },
    { "timestamp": 1718870400, "name": "revenue", "value": 1284.50 }
  ]
}

Verwandte Reihen zusammenzuhalten ist praktisch und ermöglicht es Ihnen, sie später gemeinsam zu prognostizieren. In einem Datenprofil wählen Sie aus, welche dieser Namen tatsächlich gelesen werden.

Aktualisierungen und Backfills

Der Upsert-Schlüssel ist das Paar (timestamp, name). Wird ein Record gesendet, dessen Zeitstempel und Name bereits existieren, überschreibt dies den gespeicherten Wert, statt ein Duplikat hinzuzufügen.

Backfills und erneute Versuche sind sicher

Da Schreibvorgänge auf (timestamp, name) idempotent sind, können Sie einen Batch nach einem Netzwerkaussetzer erneut abspielen, einen falschen Messwert korrigieren oder monatelange Historie in beliebiger Reihenfolge laden — das Ergebnis ist immer derselbe finale Wert pro Bucket.

Archivieren

Wenn eine Quelle keine neuen Daten mehr sammelt, können Sie sie im Dashboard archivieren. Das Archivieren stoppt die weitere Aufnahme: Eine Anfrage, die auf eine archivierte Quelle zielt, liefert HTTP 409 "Data source is archived."

Das Archivieren ist nicht destruktiv. Bestehende Daten bleiben erhalten, und alle auf der Quelle aufgebauten Datenprofile arbeiten weiterhin gegen die bereits gespeicherten Daten.
