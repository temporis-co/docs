Wie Forecasting funktioniert

Temporis erstellt Prognosen, indem es Ihre Zeitreihe als Sequenz behandelt, die ein großes Modell liest und fortsetzt — so wie ein Sprachmodell Text fortsetzt. Diese Seite baut dieses mentale Modell auf, bevor Sie die API anfassen.

Forecasting ist Sequenzvorhersage

Ein Sprachmodell liest einen Textabschnitt und sagt voraus, was als Nächstes kommt. Temporis macht dasselbe mit Zahlen. Es liest die jüngste Historie Ihrer Reihe und setzt sie fort, Schritt für Schritt.

Das Modell hat bereits eine enorme Vielfalt an Zeitreihen gesehen — Nachfrage, die am Wochenende ansteigt, Preise, die driften, Energielast, die sich täglich zyklisch verhält. Aus all dem hat es gelernt, was üblicherweise als Nächstes kommt, gegeben das, was zuvor kam. Forecasting wird so zu einer einzigen, allgemeinen Frage: Was ist, gegeben diese Historie, eine plausible Fortsetzung?

Das ist eine andere Arbeitsweise als klassisches Forecasting. Normalerweise würden Sie eine Modellfamilie auswählen und dann deren Parameter Reihe für Reihe anpassen — Trend- und Saisonterme auswählen, sie justieren und jedes Mal neu anpassen, wenn sich die Daten verschieben. Temporis überspringt den Modellaufbau pro Reihe. Sie übergeben ihm die Historie, und es setzt die Sequenz direkt fort, indem es auf Muster zurückgreift, die es über viele Reihen hinweg gelernt hat.

Von Zahlen zu Tokens

Bevor das Modell Ihre Reihe lesen kann, werden die Rohwerte in eine Sequenz diskreter Tokens umgewandelt — und die Ausgabe des Modells wird wieder in Zahlen zurückverwandelt. Die Pipeline läuft in mehreren Stufen ab:

Resampling
Die Reihe wird bereinigt und auf ein regelmäßiges Zeitraster gelegt, sodass der Abstand zwischen den Punkten einheitlich ist.
Normalisieren
Die Werte werden in einen gemeinsamen Bereich skaliert, der aus der typischen Streuung Ihrer Daten gelernt wurde (ungefähr ihr 1.-bis-99.-Perzentil-Band), damit unterschiedliche Reihen für das Modell vergleichbar aussehen.
Patchen & komprimieren
Das Raster wird in Patches zerlegt und jeder Patch mit einer Frequenztransformation komprimiert. Ein Patch ist die Einheit, die das Modell jeweils liest und schreibt.
Quantisieren
Die komprimierten Patches werden auf ein Vokabular diskreter Tokens abgebildet — das Alphabet, das das Modell tatsächlich sieht.
Fortsetzen
Das Modell liest die Token-Sequenz und erzeugt eine Fortsetzung, genau wie ein Sprachmodell Text fortsetzt.
Dekodieren
Die erzeugten Tokens werden wieder in Zahlen umgewandelt, wobei die Normalisierung rückgängig gemacht wird, sodass die Prognose auf der realen Skala Ihrer Daten landet.

Patches sind hier die zentrale Idee: Sie legen die Lese- und Schreibeinheit des Modells fest. Das Modell bewegt sich nicht jeweils um eine einzelne Roh-Beobachtung weiter — es bewegt sich jeweils um einen Patch weiter, weshalb die Prognose in Schritten statt in Einzelpunkten voranschreitet.

Sie steuern diese Kodierung

Die Einstellungen eines Datenprofils — sein interval, patch_len und patch_spacing — bestimmen, wie Ihre Rohdaten resampelt und in die obigen Patches zerlegt werden. Über das Justieren des Profils passen Sie die Kodierung für Ihre Reihe an.

Warum eine Verteilung und nicht eine einzelne Zahl

Generierung ist Sampling. Jedes Mal, wenn das Modell die Sequenz fortsetzt, zieht es Tokens, anstatt sich auf eine feste Antwort festzulegen, sodass zwei Durchläufe auf derselben Historie zwei verschiedene Zukünfte hervorbringen können. Das ist ein Feature, kein Rauschen.

Eine einzelne Prognoseanfrage fordert mehrere Fortsetzungen auf einmal an. Der Parameter count legt fest, wie viele unabhängige Trajektorien gezogen werden; jede ist ein vollständiger, in sich konsistenter Pfad in die Zukunft. Zusammen bilden sie eine probabilistische Prognose: eine Wolke plausibler Ergebnisse statt einer einzigen Linie.

Zwei Stellschrauben formen, wie diese Ziehungen gesamplet werden. temperature legt fest, wie abenteuerlustig jeder Schritt ist — höhere Werte spreizen die Samples auseinander, niedrigere halten sie nahe am wahrscheinlichsten Pfad des Modells. top_p beschneidet den unwahrscheinlichen Rand vor dem Sampling. Jede zurückgegebene Trajektorie trägt außerdem einen loss, der bewertet, wie typisch das Modell sie fand, sodass Sie einen repräsentativen Pfad auswählen oder das Set gewichten können.

Der praktische Wert liegt in der Streuung. Statt einer einzigen Schätzung, die mit ziemlicher Sicherheit leicht daneben liegt, sehen Sie die Bandbreite an Zukünften, die das Modell für plausibel hält — genug, um Best Case, Worst Case und die tatsächliche Breite der Unsicherheit abzulesen.

Siehe Prognosen & Sampling dazu, wie Sie die Antwort lesen, und Arbeiten mit Prognose-Samples dazu, wie Sie aus einem Set von Trajektorien Intervalle gewinnen.

Wo Datenprofile ins Spiel kommen

Ein Datenprofil ist das gespeicherte Rezept, das Ihre Rohdaten an die obige Kodierung bindet. Es verweist auf eine Datenquelle, benennt die zu lesenden Reihen und legt die Stellschrauben fest, die Resampling und Patching steuern — sodass jede Prognose gegen dieses Profil Ihre Daten auf dieselbe Weise liest.

Die wichtigsten Stellschrauben sind interval (der Resampling-Bucket in Sekunden), patch_len und patch_spacing (Größe und Schrittweite jedes Patches), num_patches (wie viel Historie als Kontext gelesen wird) sowie Transformationen wie use_diffs für Reihen, die multiplikativ wachsen. Die vollständige Liste samt Hinweisen zur Auswahl der einzelnen Werte finden Sie unter Datenprofile.
