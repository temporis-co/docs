Prognosen & Sampling

Eine Prognoseanfrage liefert mehrere mögliche Zukünfte zurück, nicht nur eine. Diese Seite erklärt, wie Sie die Antwort lesen und wie die Sampling-Steuerungen sie formen. Das genaue Schema finden Sie in der Predict-Referenz.

Aufbau einer Anfrage

Sie prognostizieren gegen ein Datenprofil, nicht gegen eine rohe Datenquelle. Das Profil weiß bereits, welche Reihen es lesen und wie es sie kodieren soll, sodass eine Anfrage nur angeben muss, welches Profil und wie gesampelt werden soll. Vier Felder decken das ab:

data_profilestringErforderlich

Der Name eines Ihrer Profile, etwa hourly_orders — oder ein öffentliches Profil.

countintegerErforderlich

Wie viele unabhängige Zukunftstrajektorien gezogen werden. Mindestens 1.

temperaturenumberErforderlich

Wie stark die Samples gestreut sind. Muss größer als 0 sein; etwa 1.0 ist neutral.

top_pnumberErforderlich

Nucleus-Sampling-Grenzwert im Bereich von 0 bis 1. Niedriger schneidet den unwahrscheinlichen Rand ab.

Hier ist eine Anfrage für 20 Trajektorien gegen das Profil hourly_orders:

curl https://api.temporis.co/v1/predict \
  -H "Authorization: Bearer ts_your_token" \
  -H "Content-Type: application/json" \
  -d '{
    "data_profile": "hourly_orders",
    "count": 20,
    "temperature": 1.0,
    "top_p": 0.9
  }'
import requests

resp = requests.post(
    "https://api.temporis.co/v1/predict",
    headers={"Authorization": "Bearer ts_your_token"},
    json={
        "data_profile": "hourly_orders",
        "count": 20,
        "temperature": 1.0,
        "top_p": 0.9,
    },
)
resp.raise_for_status()
result = resp.json()
const resp = await fetch("https://api.temporis.co/v1/predict", {
  method: "POST",
  headers: {
    "Authorization": "Bearer ts_your_token",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    data_profile: "hourly_orders",
    count: 20,
    temperature: 1.0,
    top_p: 0.9,
  }),
});
const result = await resp.json();

Die vollständige Referenz — jedes Feld, jeder Statuscode und jede Fehlermeldung — finden Sie in der Referenz des Predict-Endpunkts.

Die Antwort lesen

Ein erfolgreicher Aufruf gibt HTTP 200 mit vier Feldern auf oberster Ebene zurück:

request_idstring

Ein Hex-Bezeichner für diese Anfrage. Bewahren Sie ihn für den Support und Ihre eigenen Logs auf.

data_profilestring

Gibt das angefragte Profil zurück.

predict_frominteger

Unix-Sekunden des letzten beobachteten Zeitstempels, ab dem die Prognose fortgesetzt wird. Jede Trajektorie beginnt unmittelbar nach diesem Punkt.

predictionsarray

Ein Eintrag pro gesampelter Trajektorie (Länge entspricht count). Jeder enthält ein loss und ein data-Array von Prognosezeilen.

Das Beispiel unten zeigt zwei der Trajektorien aus einer orders-Prognose. Das interval des Profils ist 3600 und sein patch_spacing ist 2, sodass die Zeilen um interval × patch_spacing = 7200 Sekunden (zwei Stunden) fortschreiten:

{
  "request_id": "9f3c1a7e2b4d",
  "data_profile": "hourly_orders",
  "predict_from": 1718888400,
  "predictions": [
    {
      "loss": 0.84,
      "data": [
        { "timestamp": 1718895600, "orders": 142.0 },
        { "timestamp": 1718902800, "orders": 137.5 }
      ]
    },
    {
      "loss": 1.07,
      "data": [
        { "timestamp": 1718895600, "orders": 151.2 },
        { "timestamp": 1718902800, "orders": 129.8 }
      ]
    }
  ]
}

Jede Zeile hat einen timestamp plus ein Feld pro Reihe, die das Profil liest — hier nur orders. Die Zeitstempel schreiten um interval × patch_spacing Sekunden fort, mit demselben Abstand für jede Trajektorie, sodass Sie Samples Zeile für Zeile ausrichten können.

Die Sampling-Steuerungen

Drei Parameter formen die Menge der Zukünfte, die Sie zurückerhalten. Sie wägen Auflösung und Vielfalt gegen Kosten und Stabilität ab.

ParameterWas er bewirktTypischer WertErhöhen / senken, wenn
countAnzahl der unabhängig gezogenen Trajektorien. Mehr Samples bilden die Verteilung besser ab.20 bis 100Erhöhen für glattere Intervalle; senken, um Token-Kosten zu reduzieren.
temperatureWie stark die Samples gestreut sind. Höher ist vielfältiger und unsicherer; niedriger clustert nahe am wahrscheinlichsten Pfad.~1.0Erhöhen, wenn Prognosen zu eng wirken; senken, wenn sie unrealistisch streuen.
top_pNucleus-Grenzwert. Schneidet den unwahrscheinlichen Rand vor dem Sampling ab.0.9 bis 1.0Senken, um überraschende Sprünge zu unterdrücken; in Richtung 1.0 erhöhen, um die vollständige Verteilung zu behalten.
Kosten skalieren mit count

Jedes Sample wird unabhängig generiert und pro Token der Modellnutzung abgerechnet, sodass eine Anfrage mit count 100 etwa fünfmal so viel kostet wie eine mit count 20. Wählen Sie count so, dass die benötigte Auflösung gegen die Kosten abgewogen wird.

Loss: Trajektorien einordnen

Jede Prognose enthält ein loss. Niedriger bedeutet, dass das Modell diese Trajektorie als typischer ansah — eher im Einklang mit dem, was es angesichts der Historie erwartete. Pfade mit höherem Loss sind weiterhin gültige Samples; sie sind nur weniger zentral.

Nutzen Sie loss, um eine einzelne repräsentative Trajektorie auszuwählen, oder als Gewicht, wenn Sie die Menge zusammenfassen. Um den typischsten Pfad zu greifen:

best = min(result["predictions"], key=lambda p: p["loss"])

Multivariate Prognosen

Wenn ein Profil mehrere Reihennamen liest, enthält jede Prognosezeile ein Feld pro Name — die Reihen werden gemeinsam vorhergesagt, sodass ihre Werte innerhalb einer Zeile konsistent bleiben. Ein Profil, das kw und price gemeinsam liest, gibt Zeilen wie diese zurück:

{ "timestamp": 1718895600, "kw": 482.0, "price": 31.7 }

Öffentliche Datenprofile

Sie können gegen gemeinsam genutzte Profile prognostizieren, die Temporis pflegt, ganz ohne eigene Einrichtung — übergeben Sie einfach den öffentlichen Namen als data_profile:

{ "data_profile": "temporis/polymarket", "count": 20, "temperature": 1.0, "top_p": 0.9 }

Die Form der Antwort ist identisch; die Reihenfelder in jeder Zeile entsprechen dem, was das öffentliche Profil liest.
