Daten vorbereiten

Gute Prognosen beginnen mit sauberen, konsistenten Eingaben. Diese Anleitung beschreibt, wie Sie Ihre Historie in einer Form nach Temporis bringen, die es gut lesen kann.

Konsistent Unix-Sekunden verwenden

Jeder Zeitstempel, den Sie senden, ist ein ganzzahliger Zähler von Unix-Sekunden — die Anzahl der Sekunden seit dem 01.01.1970 00:00:00 UTC. Keine Millisekunden, kein ISO-Datumsstring. Wenn Ihr Quellsystem Millisekunden liefert, teilen Sie vor dem Senden durch 1000.

Legen Sie eine einheitliche Zeitzonen-Disziplin fest und weichen Sie nie davon ab. Die einfachste Regel, die immer funktioniert: speichern Sie alles in UTC. Unix-Sekunden sind von Natur aus UTC. Solange Sie also jeden lokalen Zeitstempel vor dem Senden in seinen tatsächlichen UTC-Moment umwandeln, bleibt Ihre Reihe in sich konsistent. Werden lokale und UTC-Werte in derselben Reihe vermischt, verschieben sich Punkte unbemerkt und das tägliche Muster, das das Modell lernt, wird verfälscht.

Ihre Zeitstempel müssen nicht auf runde Werte fallen. Das interval eines Datenprofils rastert Beobachtungen auf ein regelmäßiges Gitter — für das Profil hourly_orders wird jeder Punkt auf die nächste Stunde gebündelt. Sie liefern den tatsächlichen Moment, zu dem jede Beobachtung stattfand; das Profil übernimmt die Ausrichtung.

Eine schnelle Umrechnung

In Python liefert Ihnen int(datetime(2026, 6, 1, tzinfo=timezone.utc).timestamp()) die Unix-Sekunden für einen expliziten UTC-Moment. Hängen Sie immer eine Zeitzone an, damit die Umrechnung eindeutig ist.

Historie in Batches senden

Sie haben selten nur eine einzige Beobachtung zu senden — Sie haben Monate oder Jahre davon. Der Ingest-Endpunkt nimmt viele Datensätze pro Anfrage entgegen. Teilen Sie Ihr Backfill daher in Batches von einigen Tausend Datensätzen auf und senden Sie diese in einer Schleife, statt Datensatz für Datensatz.

Das Einlesen ist idempotent. Der Upsert-Schlüssel ist (timestamp, name). Das erneute Senden eines bereits gesendeten Datensatzes überschreibt ihn also einfach mit demselben Wert — ein No-op. Das bedeutet, dass ein Backfill sicher erneut ausgeführt werden kann: Wenn ein Batch auf halbem Weg fehlschlägt oder Ihr Skript abstürzt, starten Sie es einfach neu. Sie erzeugen keine Duplikate.

import os, requests

token = os.environ["TEMPORIS_TOKEN"]
BASE = "https://api.temporis.co"

# history is a list of (unix_seconds, value) pairs, oldest first
def records_for(history):
    return [{"timestamp": ts, "name": "orders", "value": v} for ts, v in history]

def chunked(seq, size):
    for i in range(0, len(seq), size):
        yield seq[i:i + size]

records = records_for(history)
for batch in chunked(records, 2000):
    resp = requests.post(
        f"{BASE}/v1/data_sources/ingest",
        headers={"Authorization": f"Bearer {token}"},
        json={"data_source": "store_metrics", "records": batch},
    )
    resp.raise_for_status()  # 204 No Content on success
print(f"ingested {len(records)} records")

Ein erfolgreiches Einlesen gibt HTTP 204 No Content mit leerem Body zurück, es gibt also nichts zu parsen — bestätigen Sie einfach den Status. Die vollständige Anfrageform finden Sie in der Ingest-Referenz.

Dem Modell genug Historie geben

Ein Profil kann keine Prognosen liefern, bis es bereit ist — Temporis benötigt genügend Historie, um die Normalisierungsstatistiken zu berechnen, durch die es Ihre Reihe liest. Und jede Prognose benötigt mindestens min_patches an aktuellem Kontext, sonst gibt die Anfrage 409 "Not enough data for this data profile." zurück.

Die praktische Regel: im Zweifel mehr Historie senden. Mehr Historie ermöglicht es einem Profil, bereit zu werden, überschreitet min_patches komfortabel und gibt dem Modell mehr saisonale Zyklen zum Lernen. Es gibt keinen Nachteil dabei, mehr zu senden, als das Kontextfenster nutzt — ältere Punkte jenseits von num_patches werden zur Prognosezeit einfach nicht gelesen.

Was min_patches, num_patches und "bereit" bedeuten und wie sie mit Ihrem interval zusammenhängen, erfahren Sie unter Datenprofile.

Unregelmäßige Zeitstempel und Lücken

Echte Daten sind unordentlich. Sensoren überspringen eine Messung, ein Job läuft ein paar Minuten zu spät, ein Geschäft ist an einem Feiertag geschlossen. Sie müssen nichts davon von Hand bereinigen — der Resampling-Schritt des Profils übernimmt das.

Das interval bündelt Ihre Beobachtungen auf ein regelmäßiges Gitter, was leicht unregelmäßige Zeitstempel abfängt: Eine Messung, die um 10:58 eintrifft, und eine um 11:03 landen beide im 11:00-Bucket. Für fehlende Buckets legt fill_limit fest, wie viele aufeinanderfolgende leere Buckets Temporis per Forward-Fill auffüllt, bevor es aufgibt. Eine kurze Lücke wird überbrückt; eine lange Lücke bleibt ein Loch, und diese Zeilen werden verworfen, statt erfunden.

Wenn Ihre Reihe wiederkehrende Totzeiten hat — Stunden, in denen unmöglich etwas passieren kann — behandeln Sie sie gar nicht erst als fehlende Daten. Ein exclude_window maskiert diesen Tageszeitbereich, sodass er die Reihe nie verzerrt. Das ist eine Profileinstellung, behandelt unter Datenprofile.

Sie senden die Rohbeobachtungen so, wie Sie sie haben. Wo sie landen, wie Lücken gefüllt werden und welche Totzeit ignoriert wird, sind Entscheidungen, die das Profil trifft — keine Transformationen, die Sie fest in die Daten einbauen.

Mehrere Reihen ausrichten

Ein Profil kann mehrere Reihen gleichzeitig lesen — ein multivariates Profil, das sie gemeinsam prognostiziert. Damit das gut funktioniert, brauchen die Reihen eine gemeinsame Zeitachse: Senden Sie die Beobachtung jeder Reihe zu denselben Zeitstempeln, damit sich die Zeilen nach dem Bucketing decken.

Sie können mehrere Reihen in einer einzigen Ingest-Anfrage senden — fügen Sie einfach pro Reihe einen Datensatz zu jedem Zeitstempel hinzu. Hier teilen sich zwei Reihen, orders und visitors, dieselbe Zeitachse:

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718841600, "name": "orders",   "value": 42 },
    { "timestamp": 1718841600, "name": "visitors", "value": 530 },
    { "timestamp": 1718845200, "name": "orders",   "value": 39 },
    { "timestamp": 1718845200, "name": "visitors", "value": 504 },
    { "timestamp": 1718848800, "name": "orders",   "value": 51 },
    { "timestamp": 1718848800, "name": "visitors", "value": 612 }
  ]
}

Wenn eine Reihe zu einem Zeitstempel fehlt, an dem eine andere vorhanden ist, wird dieser Bucket gemäß fill_limit gefüllt, genau wie jede andere Lücke — ein paar Nachzügler brechen die Ausrichtung also nicht, aber Sie erhalten die sauberste gemeinsame Prognose, wenn sich die Reihen auf derselben Uhr bewegen.

Korrekturen und Backfills

Einen falschen Wert gefunden? Ein Quartal neu darstellen? Sie löschen oder patchen nichts Besonderes — Sie senden einfach den korrigierten Datensatz erneut. Da der Upsert-Schlüssel (timestamp, name) ist, überschreibt das Senden eines neuen value zu einem bestehenden timestamp+name den alten.

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718841600, "name": "orders", "value": 47 }
  ]
}
Erneutes Ausführen hat keine Nebenwirkungen

Da jedes Einlesen ein idempotenter Upsert ist, kann sich ein korrigiertes Backfill sicher mit bereits gesendeten Daten überlappen. Senden Sie den gesamten betroffenen Bereich erneut, wenn das einfacher ist, als genau zu berechnen, welche Punkte sich geändert haben.
