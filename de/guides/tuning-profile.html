Ein Datenprofil abstimmen

Die Standardwerte funktionieren, aber ein Profil auf den Rhythmus Ihrer Daten abzustimmen macht Prognosen spürbar besser. So wählen Sie jede Einstellung. (Die Einstellungen sind unter Datenprofile definiert.)

Intervall an Ihren Takt anpassen

Die mit Abstand wichtigste Einstellung ist interval — der Bucket in Sekunden, auf den Ihre Beobachtungen neu abgetastet werden. Setzen Sie ihn auf die tatsächliche Abtastperiode Ihrer Daten. Wenn Sie Bestellungen einmal pro Stunde erfassen, verwenden Sie 3600. Wenn Sie eine Zahl pro Tag haben, verwenden Sie 86400.

Liegen Sie in einer der beiden Richtungen falsch, hat das Modell Probleme:

  • Zu fein — ein Intervall, das kürzer ist als Ihr tatsächlicher Takt, polstert die Reihe mit aufgefüllten Buckets auf und verbraucht das Kontextfenster für Rauschen und Leerraum statt für Signal.
  • Zu grob — ein Intervall, das länger ist als Ihr Takt, mittelt echte Struktur weg. Ein stündliches Nachfragemuster verschwindet einfach, wenn Sie es auf einen Tag bündeln.

Das Profil hourly_orders verwendet interval: 3600, weil store_metrics eine Bestellzahl pro Stunde erfasst. Beginnen Sie damit, die Periode anzupassen, in der Ihre Daten tatsächlich eintreffen.

Wie viel Historie: num_patches und min_patches

Diese beiden Einstellungen steuern das Maximum und das Minimum an Historie, das in eine Prognose einfließt.

num_patches ist das Kontextfenster: die maximale Anzahl an Patches aktueller Historie, die das Modell liest. Mehr Patches bedeuten mehr vergangene Zyklen im Blick, also mehr Saisonalität, die das Modell aufnehmen kann — erhöhen Sie es, wenn es wöchentliche oder jährliche Struktur "sehen" soll. Der Nutzen flacht jedoch ab; ab einem bestimmten Punkt füttern Sie alte Historie ein, die die Gegenwart nicht mehr widerspiegelt, für mehr Tokens und keine bessere Prognose.

min_patches ist die Untergrenze: die Mindesthistorie, die ein Profil benötigt, bevor es überhaupt liefert. Darunter gibt ein Predict-Aufruf 409 "Not enough data for this data profile." zurück. Erhöhen Sie min_patches, wenn Sie eine Prognose mit geringer Konfidenz lieber ablehnen, als eine zu liefern, die auf einer Handvoll Punkten beruht — nützlich für eine brandneue Reihe, bei der eine frühe Schätzung irreführend wäre.

Stellen Sie es sich als Spanne vor: min_patches ist "versuche es darunter gar nicht erst" und num_patches ist "blicke nicht weiter zurück als bis hierhin".

Auflösung vs. Horizont: patch_len und patch_spacing

Ein Patch ist der Block an Punkten, den das Modell jeweils liest und schreibt. patch_len legt fest, wie viele Punkte in diesem Block sind. patch_spacing legt den Schritt zwischen den Punkten fest, die Sie tatsächlich verwenden — es tastet die Reihe herunter und dehnt dabei, wie weit jeder erzeugte Schritt in die Zukunft reicht.

Die zentrale Beziehung: Der effektive Zeitschritt zwischen Prognosepunkten beträgt interval * patch_spacing Sekunden. Mit interval: 3600 und patch_spacing: 1 ist jeder Schritt eine Stunde. Erhöhen Sie patch_spacing auf 3 und jeder Schritt umfasst drei Stunden — dieselbe Anzahl erzeugter Punkte reicht nun dreimal so weit in die Zukunft, bei gröberer Auflösung.

Dies ist also ein Kompromiss zwischen Detailgrad und Reichweite. Halten Sie das Spacing eng, wenn Sie feingranulare kurzfristige Prognosen brauchen; weiten Sie es, wenn Ihnen ein längerer Horizont wichtig ist und Sie einen gröberen Schritt akzeptieren können.

Niveaus vs. Wachstum: use_diffs

use_diffs (die Log-Diff-Transformation) weist das Modell an, mit relativer Veränderung zu arbeiten — logarithmisches Wachstum von Punkt zu Punkt — statt mit dem rohen Niveau.

  • Schalten Sie es ein für Reihen, die multiplikativ wachsen oder sich verstärken: Umsatz, Preise, Nutzerzahlen, alles mit einem Trend oder das sich "verdoppelt" statt "addiert". Wachstum zu modellieren verhindert, dass eine steigende Reihe an ihrem früheren Stand verankert bleibt.
  • Lassen Sie es aus für begrenzte oder stationäre Niveaus, die in einem festen Bereich schwanken: Auslastungsprozentsatz, Temperatur, eine Sättigungsrate. Diese haben ein natürliches Niveau, zu dem sie zurückkehren, und das Differenzieren fügt ihnen nur Rauschen hinzu.

Ein schneller Test: Wenn eine Verdopplung der Einheiten Ihrer Reihe auch deren typische Sprünge verdoppeln sollte, ist sie multiplikativ — schalten Sie use_diffs ein.

Totzeit maskieren: Exclude-Windows

Manche Reihen haben wiederkehrende Abschnitte, in denen per Design nichts passiert — ein Markt, der über Nacht geschlossen ist, ein Geschäft außerhalb der Öffnungszeiten. Wenn Sie diese Totzeit drin lassen, behandelt das Modell diese flachen oder Null-Abschnitte als echtes Signal und lernt ein Muster, das gar nicht da ist.

Ein exclude_window (eine Zeitzone plus eine Start- und Stopp-Tageszeit) maskiert diesen wiederkehrenden Bereich, sodass er aus der Reihe, die das Modell liest, entfernt wird. Setzen Sie eines für die Stunden, in denen Ihre Reihe strukturell untätig ist — etwa das Maskieren der Nacht bei einem Profil, das Marktpreise liest — damit das Verhalten während der Öffnungszeiten sauber bleibt.

Lücken tolerieren: fill_limit

fill_limit ist die längste Folge aufeinanderfolgender fehlender Buckets, die Temporis per Forward-Fill auffüllt. Eine Lücke, die kürzer ist als das Limit, wird durch Übertragen des letzten Werts überbrückt; eine Lücke, die länger ist als das Limit, bleibt ein Loch, und diese Zeilen werden verworfen.

  • Erhöhen Sie es, um längere Ausfälle zu überbrücken — ein Sensor, der gelegentlich für eine Weile offline geht, wo es vernünftig ist, die letzte Messung weiterzuführen.
  • Senken Sie es, wenn Sie über große Löcher hinweg lieber keine Daten erfinden möchten — wenn eine lange Lücke wirklich "unbekannt" bedeutet und eine flache, weitergeführte Linie irreführend wäre.

Ändern Sie jeweils nur eine Einstellung und vergleichen Sie die Prognosen. Das Abstimmen ist leichter nachzuvollziehen, wenn jede Anpassung eine sichtbare, isolierte Wirkung hat.
