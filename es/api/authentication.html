Autenticación

Cada solicitud a la API de Temporis se autentica con un token de portador. Aquí te explicamos cómo crear uno, enviarlo y mantenerlo a salvo.

Tokens de acceso

Los tokens de acceso se crean en el panel, en Tokens de acceso. Un token es una cadena aleatoria larga con el prefijo ts_.

El token completo se muestra una sola vez, en el momento en que lo creas. Temporis almacena únicamente un hash del mismo en el servidor, nunca el token en sí. Esto significa que un token perdido no se puede recuperar: si lo pierdes, elimínalo y crea uno nuevo.

Trata los tokens como contraseñas

Cualquiera que tenga uno de tus tokens puede ingerir datos y ejecutar predicciones, y las predicciones se facturan a tu cuenta. Protégelos en consecuencia.

Autenticar una solicitud

Envía el token en la cabecera Authorization en cada solicitud, usando el esquema Bearer:

Authorization: Bearer ts_your_token_here

Los ejemplos a lo largo de esta documentación leen el token de una variable de entorno TEMPORIS_TOKEN, de modo que el secreto nunca aparece en el código fuente. Expórtalo una vez en tu shell:

export TEMPORIS_TOKEN="ts_your_token_here"

Luego adjúntalo como cabecera:

curl https://api.temporis.co/v1/predict \
  -H "Authorization: Bearer $TEMPORIS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{ "data_profile": "hourly_orders", "count": 5, "temperature": 1.0, "top_p": 0.9 }'
import os, requests

token = os.environ["TEMPORIS_TOKEN"]

resp = requests.post(
    "https://api.temporis.co/v1/predict",
    headers={"Authorization": f"Bearer {token}"},
    json={"data_profile": "hourly_orders", "count": 5, "temperature": 1.0, "top_p": 0.9},
)
resp.raise_for_status()
const token = process.env.TEMPORIS_TOKEN;

const resp = await fetch("https://api.temporis.co/v1/predict", {
  method: "POST",
  headers: {
    "Authorization": `Bearer ${token}`,
    "Content-Type": "application/json",
  },
  body: JSON.stringify({ data_profile: "hourly_orders", count: 5, temperature: 1.0, top_p: 0.9 }),
});

Cuando la autenticación falla

Si la cabecera Authorization falta o está mal formada, o el token es desconocido, la API responde con HTTP 401 y un cuerpo JSON que describe qué salió mal:

{ "detail": "Invalid access token." }

Una cabecera ausente o mal formada devuelve en su lugar { "detail": "Missing access token." }. Ambos son 401. Consulta Errores y códigos de estado para ver la lista completa de respuestas.

Mantener los tokens a salvo

Unos cuantos hábitos mantienen tus tokens fuera de las manos equivocadas:

  • Nunca incluyas tokens en el control de versiones, y nunca los pongas en código del lado del cliente. Los navegadores y las aplicaciones móviles entregan su código fuente a los usuarios.
  • Lee los tokens desde variables de entorno o un gestor de secretos. Mantenlos fuera de los registros, capturas de pantalla y tickets.
  • Crea un token independiente por servicio para poder revocar uno sin afectar a los demás.
  • Elimina un token en el panel para revocarlo al instante. La eliminación surte efecto de inmediato: la siguiente solicitud que use ese token recibe un 401.
