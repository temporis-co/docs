Errores y códigos de estado

Cómo Temporis informa el éxito y el fallo, qué significa cada estado y cómo manejarlos.

Forma de la respuesta

La forma de una respuesta depende de lo que haya ocurrido:

  • Una ingesta exitosa devuelve 204 No Content sin cuerpo.
  • Una predicción exitosa devuelve 200 con un cuerpo JSON.
  • Cualquier error devuelve el estado HTTP correspondiente con un cuerpo JSON con la forma { "detail": "..." }.
{ "detail": "Data profile not found." }

Manejo de errores

La mayoría de los fallos se asignan a una siguiente acción clara:

  • 401 — compruebe que el encabezado Authorization esté presente y que el token sea válido. Consulte Autenticación.
  • 402 — agregue o corrija una tarjeta de crédito en el panel, en la sección Facturación, y vuelva a intentarlo.
  • 404 — asegúrese de que la fuente de datos o el perfil de datos exista y de que el perfil esté listo para servir.
  • 409 en predict — ingiera más historial hasta que el perfil alcance su mínimo, y vuelva a intentarlo.
  • 409 en ingest — la fuente de datos está archivada; desarchívela o escriba en otra fuente.
  • 502 — transitorio. Reintente con backoff exponencial.

Una simple ramificación según el código de estado cubre todos estos casos:

import time, requests

def predict(payload, token, retries=3):
    for attempt in range(retries):
        resp = requests.post(
            "https://api.temporis.co/v1/predict",
            headers={"Authorization": f"Bearer {token}"},
            json=payload,
        )
        if resp.status_code == 200:
            return resp.json()
        if resp.status_code == 502:
            time.sleep(2 ** attempt)   # transient: back off and retry
            continue
        detail = resp.json().get("detail", "")
        if resp.status_code == 402:
            raise RuntimeError(f"Billing: {detail} — add a card in the dashboard.")
        if resp.status_code == 409:
            raise RuntimeError(f"Not ready: {detail} — ingest more history.")
        resp.raise_for_status()        # 400 / 401 / 404: fix the request
    raise RuntimeError("Prediction failed after retries.")

Reintentos

Reintente lo correcto

Un 502 es seguro de reintentar — es transitorio. La ingesta es idempotente (las escrituras hacen upsert sobre la clave (timestamp, name)), por lo que reintentar una ingesta fallida siempre es seguro. Evite reintentar a ciegas las respuestas 4xx; corrija primero la solicitud.
