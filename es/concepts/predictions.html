Predicciones y muestreo

Una solicitud de predicción devuelve varios futuros posibles en lugar de uno solo. Esta página explica cómo leer la respuesta y cómo los controles de muestreo la moldean. Para el esquema exacto, consulta la referencia de predict.

Anatomía de una solicitud

Predices contra un perfil de datos, no contra una fuente de datos en bruto. El perfil ya sabe qué series leer y cómo codificarlas, así que una solicitud solo tiene que indicar qué perfil y cómo muestrear. Cuatro campos lo cubren:

data_profilestringObligatorio

El nombre de uno de tus perfiles, como hourly_orders — o un perfil público.

countintegerObligatorio

Cuántas trayectorias futuras independientes generar. Al menos 1.

temperaturenumberObligatorio

Cuán dispersas están las muestras. Debe ser mayor que 0; alrededor de 1.0 es neutral.

top_pnumberObligatorio

Umbral de muestreo de núcleo (nucleus) en el rango de 0 a 1. Más bajo recorta la cola improbable.

Aquí tienes una solicitud de 20 trayectorias contra el perfil hourly_orders:

curl https://api.temporis.co/v1/predict \
  -H "Authorization: Bearer ts_your_token" \
  -H "Content-Type: application/json" \
  -d '{
    "data_profile": "hourly_orders",
    "count": 20,
    "temperature": 1.0,
    "top_p": 0.9
  }'
import requests

resp = requests.post(
    "https://api.temporis.co/v1/predict",
    headers={"Authorization": "Bearer ts_your_token"},
    json={
        "data_profile": "hourly_orders",
        "count": 20,
        "temperature": 1.0,
        "top_p": 0.9,
    },
)
resp.raise_for_status()
result = resp.json()
const resp = await fetch("https://api.temporis.co/v1/predict", {
  method: "POST",
  headers: {
    "Authorization": "Bearer ts_your_token",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    data_profile: "hourly_orders",
    count: 20,
    temperature: 1.0,
    top_p: 0.9,
  }),
});
const result = await resp.json();

La referencia completa — cada campo, código de estado y mensaje de error — está en la referencia del endpoint de predict.

Lectura de la respuesta

Una llamada exitosa devuelve HTTP 200 con cuatro campos de nivel superior:

request_idstring

Un identificador hexadecimal para esta solicitud. Consérvalo para soporte y tus propios registros.

data_profilestring

Repite el perfil que solicitaste.

predict_frominteger

Segundos Unix de la última marca de tiempo observada desde la que continúa el pronóstico. Cada trayectoria comienza justo después de este punto.

predictionsarray

Una entrada por trayectoria muestreada (la longitud es igual a count). Cada una lleva un loss y un array data de filas de pronóstico.

El ejemplo de abajo muestra dos de las trayectorias de un pronóstico de orders. El interval del perfil es 3600 y su patch_spacing es 2, así que las filas avanzan de interval × patch_spacing = 7200 segundos (dos horas):

{
  "request_id": "9f3c1a7e2b4d",
  "data_profile": "hourly_orders",
  "predict_from": 1718888400,
  "predictions": [
    {
      "loss": 0.84,
      "data": [
        { "timestamp": 1718895600, "orders": 142.0 },
        { "timestamp": 1718902800, "orders": 137.5 }
      ]
    },
    {
      "loss": 1.07,
      "data": [
        { "timestamp": 1718895600, "orders": 151.2 },
        { "timestamp": 1718902800, "orders": 129.8 }
      ]
    }
  ]
}

Cada fila tiene un timestamp más un campo por serie que el perfil lee — aquí solo orders. Las marcas de tiempo avanzan de interval × patch_spacing segundos, el mismo espaciado para cada trayectoria, de modo que puedes alinear las muestras fila por fila.

Los controles de muestreo

Tres parámetros moldean el conjunto de futuros que recibes. Equilibran resolución y diversidad frente a coste y estabilidad.

ParámetroQué haceValor típicoSúbelo / bájalo cuando
countNúmero de trayectorias independientes generadas. Más muestras resuelven mejor la distribución.20 a 100Súbelo para intervalos más suaves; bájalo para reducir el coste en tokens.
temperatureCuán dispersas están las muestras. Más alto es más diverso e incierto; más bajo se agrupa cerca del camino más probable.~1.0Súbelo cuando los pronósticos se vean demasiado estrechos; bájalo cuando se dispersen de forma poco realista.
top_pUmbral de núcleo (nucleus). Recorta la cola improbable antes de muestrear.0.9 a 1.0Bájalo para suprimir saltos sorprendentes; súbelo hacia 1.0 para conservar la distribución completa.
El coste escala con count

Cada muestra se genera de forma independiente y se mide por token de uso del modelo, así que una solicitud con count 100 cuesta aproximadamente cinco veces más que una con count 20. Elige count para equilibrar la resolución que necesitas frente al coste.

Loss: ordenar las trayectorias

Cada predicción lleva un loss. Más bajo significa que el modelo consideró esa trayectoria más típica — más acorde con lo que esperaba dado el histórico. Los caminos con mayor loss siguen siendo muestras válidas; simplemente son menos centrales.

Usa loss para elegir una única trayectoria representativa, o como peso cuando resumas el conjunto. Para tomar el camino más típico:

best = min(result["predictions"], key=lambda p: p["loss"])

Predicciones multivariantes

Si un perfil lee varios nombres de series, cada fila de pronóstico lleva un campo por nombre — las series se predicen conjuntamente, de modo que sus valores se mantienen consistentes dentro de una fila. Un perfil que lee kw y price juntos devuelve filas como:

{ "timestamp": 1718895600, "kw": 482.0, "price": 31.7 }

Perfiles de datos públicos

Puedes predecir contra perfiles compartidos que Temporis mantiene, sin ninguna configuración propia — simplemente pasa el nombre público como data_profile:

{ "data_profile": "temporis/polymarket", "count": 20, "temperature": 1.0, "top_p": 0.9 }

La forma de la respuesta es idéntica; los campos de series en cada fila coinciden con lo que sea que el perfil público lea.
