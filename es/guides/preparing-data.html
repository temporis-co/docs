Preparar tus datos

Los buenos pronósticos comienzan con una entrada limpia y consistente. Esta guía explica cómo introducir tu histórico en Temporis con una forma que pueda leer bien.

Usa segundos Unix, de forma consistente

Cada marca de tiempo que envías es un recuento entero de segundos Unix: el número de segundos desde 1970-01-01 00:00:00 UTC. No milisegundos, ni una cadena de fecha ISO. Si tu sistema de origen te da milisegundos, divide entre 1000 antes de enviar.

Elige una única disciplina de zona horaria y no te desvíes nunca de ella. La regla más sencilla que siempre funciona: almacena todo en UTC. Los segundos Unix son intrínsecamente UTC, así que mientras conviertas cada marca de tiempo local a su instante UTC real antes de enviar, tu serie se mantiene internamente consistente. Mezclar valores en hora local y en UTC en la misma serie desplaza puntos de forma silenciosa y corrompe el patrón diario que el modelo aprende.

Tus marcas de tiempo no necesitan caer en números redondos. El interval de un perfil de datos encaja las observaciones en una rejilla regular: en el perfil hourly_orders, cada punto se agrupa en el cubo de la hora más cercana. Tú aportas el instante real en que ocurrió cada observación; el perfil se encarga del alineamiento.

Una conversión rápida

En Python, int(datetime(2026, 6, 1, tzinfo=timezone.utc).timestamp()) te da los segundos Unix de un instante UTC explícito. Adjunta siempre una zona horaria para que la conversión no sea ambigua.

Envía el histórico en lotes

Rara vez tienes una sola observación que enviar: tienes meses o años de ellas. El endpoint de ingesta acepta muchos registros por petición, así que divide tu carga histórica en lotes de unos pocos miles de registros y publícalos en un bucle, en lugar de un registro a la vez.

La ingesta es idempotente. La clave de upsert es (timestamp, name), así que reenviar un registro que ya enviaste simplemente lo sobrescribe con el mismo valor: una operación sin efecto. Eso significa que una carga histórica es segura de reejecutar: si un lote falla a medias o tu script se cae, basta con volver a iniciarlo. No crearás duplicados.

import os, requests

token = os.environ["TEMPORIS_TOKEN"]
BASE = "https://api.temporis.co"

# history is a list of (unix_seconds, value) pairs, oldest first
def records_for(history):
    return [{"timestamp": ts, "name": "orders", "value": v} for ts, v in history]

def chunked(seq, size):
    for i in range(0, len(seq), size):
        yield seq[i:i + size]

records = records_for(history)
for batch in chunked(records, 2000):
    resp = requests.post(
        f"{BASE}/v1/data_sources/ingest",
        headers={"Authorization": f"Bearer {token}"},
        json={"data_source": "store_metrics", "records": batch},
    )
    resp.raise_for_status()  # 204 No Content on success
print(f"ingested {len(records)} records")

Una ingesta exitosa devuelve HTTP 204 No Content con un cuerpo vacío, así que no hay nada que parsear: solo confirma el estado. Consulta la referencia de ingesta para ver la forma completa de la petición.

Dale al modelo suficiente histórico

Un perfil no puede servir predicciones hasta que esté listo: Temporis necesita suficiente histórico para calcular las estadísticas de normalización a través de las cuales lee tu serie. Y cada predicción necesita al menos min_patches de contexto reciente, o la petición devuelve 409 "Not enough data for this data profile."

La regla práctica: ante la duda, envía más histórico. Más histórico permite que un perfil esté listo, supera holgadamente min_patches y le da al modelo más ciclos estacionales de los que aprender. No hay penalización por enviar más de lo que usa la ventana de contexto: los puntos más antiguos que num_patches simplemente no se leen en el momento de la predicción.

Para entender qué significan min_patches, num_patches y "listo" y cómo se relacionan con tu interval, lee Perfiles de datos.

Marcas de tiempo irregulares y huecos

Los datos reales son caóticos. Un sensor se salta una lectura, un trabajo se ejecuta con unos minutos de retraso, una tienda cierra en un día festivo. No necesitas limpiar nada de esto a mano: el paso de remuestreo del perfil se encarga de ello.

El interval agrupa tus observaciones en una rejilla regular, lo que absorbe marcas de tiempo ligeramente irregulares: una lectura que llega a las 10:58 y otra a las 11:03 caen ambas en el cubo de las 11:00. Para los cubos faltantes, fill_limit establece cuántos cubos vacíos consecutivos rellenará Temporis hacia adelante antes de rendirse. Un hueco corto se salva; un hueco largo se deja como un agujero y esas filas se descartan en lugar de inventarse.

Si tu serie tiene tiempo muerto recurrente —horas en las que no puede pasar nada— no lo trates como datos faltantes en absoluto. Un exclude_window enmascara ese rango horario para que nunca distorsione la serie. Esa es una opción del perfil, tratada en Perfiles de datos.

Envías las observaciones en bruto tal como las tienes. Dónde caen, cómo se rellenan los huecos y qué tiempo muerto se ignora son decisiones que toma el perfil, no transformaciones que cocines dentro de los datos.

Alinear múltiples series

Un perfil puede leer varias series a la vez: un perfil multivariante que las pronostica de forma conjunta. Para que eso funcione bien, las series necesitan una línea de tiempo común: envía la observación de cada serie en el mismo conjunto de marcas de tiempo para que las filas se alineen tras la agrupación.

Puedes enviar varias series en una sola petición de ingesta: simplemente incluye un registro por serie en cada marca de tiempo. Aquí dos series, orders y visitors, comparten la misma línea de tiempo:

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718841600, "name": "orders",   "value": 42 },
    { "timestamp": 1718841600, "name": "visitors", "value": 530 },
    { "timestamp": 1718845200, "name": "orders",   "value": 39 },
    { "timestamp": 1718845200, "name": "visitors", "value": 504 },
    { "timestamp": 1718848800, "name": "orders",   "value": 51 },
    { "timestamp": 1718848800, "name": "visitors", "value": 612 }
  ]
}

Si una serie falta en una marca de tiempo donde otra está presente, ese cubo se rellena según fill_limit igual que cualquier otro hueco, así que unos pocos rezagados no romperán el alineamiento, pero obtendrás el pronóstico conjunto más limpio cuando las series avancen al mismo compás.

Correcciones y recargas

¿Encontraste un valor erróneo? ¿Reformulas un trimestre? No borras ni parcheas nada especial: simplemente reenvías el registro corregido. Como la clave de upsert es (timestamp, name), enviar un nuevo value en una marca de tiempo+nombre existente sobrescribe el anterior.

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718841600, "name": "orders", "value": 47 }
  ]
}
Reejecutar no tiene efectos secundarios

Como cada ingesta es un upsert idempotente, una recarga corregida puede solaparse sin riesgo con datos que ya enviaste. Vuelve a publicar todo el rango afectado si es más fácil que calcular exactamente qué puntos cambiaron.
