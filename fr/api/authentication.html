Authentification

Chaque requête de l'API Temporis s'authentifie avec un jeton bearer. Voici comment en créer un, l'envoyer et le garder en sécurité.

Jetons d'accès

Vous créez des jetons d'accès dans le tableau de bord, sous Jetons d'accès. Un jeton est une longue chaîne aléatoire préfixée par ts_.

Le jeton complet est affiché une seule fois, au moment où vous le créez. Temporis ne stocke côté serveur qu'un hachage de celui-ci, jamais le jeton lui-même. Cela signifie qu'un jeton perdu ne peut pas être récupéré — si vous le perdez, supprimez-le et créez-en un nouveau.

Traitez les jetons comme des mots de passe

Quiconque détient l'un de vos jetons peut ingérer des données et exécuter des prédictions — et les prédictions sont facturées sur votre compte. Protégez-les en conséquence.

Authentifier une requête

Envoyez le jeton dans l'en-tête Authorization de chaque requête, en utilisant le schéma Bearer :

Authorization: Bearer ts_your_token_here

Les exemples tout au long de cette documentation lisent le jeton depuis une variable d'environnement TEMPORIS_TOKEN, afin que le secret n'apparaisse jamais dans le code source. Exportez-le une fois dans votre shell :

export TEMPORIS_TOKEN="ts_your_token_here"

Puis attachez-le comme en-tête :

curl https://api.temporis.co/v1/predict \
  -H "Authorization: Bearer $TEMPORIS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{ "data_profile": "hourly_orders", "count": 5, "temperature": 1.0, "top_p": 0.9 }'
import os, requests

token = os.environ["TEMPORIS_TOKEN"]

resp = requests.post(
    "https://api.temporis.co/v1/predict",
    headers={"Authorization": f"Bearer {token}"},
    json={"data_profile": "hourly_orders", "count": 5, "temperature": 1.0, "top_p": 0.9},
)
resp.raise_for_status()
const token = process.env.TEMPORIS_TOKEN;

const resp = await fetch("https://api.temporis.co/v1/predict", {
  method: "POST",
  headers: {
    "Authorization": `Bearer ${token}`,
    "Content-Type": "application/json",
  },
  body: JSON.stringify({ data_profile: "hourly_orders", count: 5, temperature: 1.0, top_p: 0.9 }),
});

Quand l'authentification échoue

Si l'en-tête Authorization est manquant ou mal formé, ou si le jeton est inconnu, l'API répond avec HTTP 401 et un corps JSON décrivant ce qui n'a pas fonctionné :

{ "detail": "Invalid access token." }

Un en-tête manquant ou mal formé renvoie plutôt { "detail": "Missing access token." }. Les deux sont des 401. Consultez Erreurs & codes de statut pour la liste complète des réponses.

Garder les jetons en sécurité

Quelques habitudes gardent vos jetons hors de portée des mauvaises mains :

  • Ne commettez jamais de jetons dans le contrôle de version, et ne les placez jamais dans du code côté client. Les navigateurs et les applications mobiles livrent leur code source aux utilisateurs.
  • Lisez les jetons depuis des variables d'environnement ou un gestionnaire de secrets. Gardez-les hors des journaux, des captures d'écran et des tickets.
  • Créez un jeton distinct par service afin de pouvoir en révoquer un sans perturber les autres.
  • Supprimez un jeton dans le tableau de bord pour le révoquer instantanément. La suppression prend effet immédiatement — la requête suivante utilisant ce jeton reçoit un 401.
