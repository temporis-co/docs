Erreurs & codes de statut

Comment Temporis signale le succès et l'échec, ce que signifie chaque statut et comment les gérer.

Forme de la réponse

La forme d'une réponse dépend de ce qui s'est produit :

  • Une ingestion réussie renvoie 204 No Content sans corps.
  • Une prédiction réussie renvoie 200 avec un corps JSON.
  • Toute erreur renvoie le statut HTTP correspondant avec un corps JSON de la forme { "detail": "..." }.
{ "detail": "Data profile not found." }

Codes de statut

StatutSignificationOù
200 OKLa prédiction a réussi ; le corps contient les échantillons.predict
204 No ContentL'ingestion a réussi ; pas de corps.ingest
400 Bad RequestParamètres de prédiction invalides.predict
401 UnauthorizedJeton d'accès manquant ou invalide.toutes les requêtes
402 Payment RequiredAucune carte enregistrée, ou un problème de paiement.predict
404 Not FoundSource de données introuvable (ingest) ; profil de données introuvable ou pas prêt à servir (predict).ingest, predict
409 ConflictSource de données archivée (ingest) ; pas assez de données (predict).ingest, predict
502 Bad GatewayLa prédiction a échoué — une erreur côté serveur transitoire.predict

Gérer les erreurs

La plupart des échecs correspondent à une action suivante claire :

  • 401 — vérifiez que l'en-tête Authorization est présent et que le jeton est valide. Consultez Authentification.
  • 402 — ajoutez ou corrigez une carte de crédit dans le tableau de bord sous Facturation, puis réessayez.
  • 404 — assurez-vous que la source de données ou le profil de données existe, et que le profil est prêt à servir.
  • 409 sur predict — ingérez davantage d'historique jusqu'à ce que le profil atteigne son minimum, puis réessayez.
  • 409 sur ingest — la source de données est archivée ; désarchivez-la ou écrivez dans une autre source.
  • 502 — transitoire. Réessayez avec un délai exponentiel.

Un simple branchement sur le code de statut couvre tous ces cas :

import time, requests

def predict(payload, token, retries=3):
    for attempt in range(retries):
        resp = requests.post(
            "https://api.temporis.co/v1/predict",
            headers={"Authorization": f"Bearer {token}"},
            json=payload,
        )
        if resp.status_code == 200:
            return resp.json()
        if resp.status_code == 502:
            time.sleep(2 ** attempt)   # transient: back off and retry
            continue
        detail = resp.json().get("detail", "")
        if resp.status_code == 402:
            raise RuntimeError(f"Billing: {detail} — add a card in the dashboard.")
        if resp.status_code == 409:
            raise RuntimeError(f"Not ready: {detail} — ingest more history.")
        resp.raise_for_status()        # 400 / 401 / 404: fix the request
    raise RuntimeError("Prediction failed after retries.")

Nouvelles tentatives

Réessayez les bonnes choses

Un 502 peut être réessayé en toute sécurité — il est transitoire. L'ingestion est idempotente (les écritures font un upsert sur la clé (timestamp, name)), donc réessayer une ingestion échouée est toujours sans danger. Évitez de réessayer aveuglément les réponses 4xx ; corrigez d'abord la requête.
