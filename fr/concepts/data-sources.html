Sources de données

Une source de données est l'endroit où vivent vos observations brutes. Cette page couvre le modèle de données de Temporis et comment y envoyer des données correctement.

Vous créez une source de données une seule fois dans le tableau de bord, puis vous y diffusez des observations via l'API au fur et à mesure qu'elles se produisent.

Le modèle de données

Temporis stocke les données sous forme de lignes. Chaque observation est un simple triplet : quand quelque chose a été mesuré, quoi a été mesuré et le nombre que vous avez mesuré.

timestamp
Temps Unix en secondes (un entier). Le moment où l'observation a été prise.
name
La série à laquelle cette valeur appartient — par exemple orders ou revenue.
value
Un nombre. Les nombres à virgule flottante sont acceptés.

Une seule source de données peut contenir de nombreuses séries nommées côte à côte. Le champ name est ce qui les distingue. Vos données sont isolées de celles de tout autre client — une source de données vous appartient à vous seul.

Créer une source de données

Vous créez une source de données dans le tableau de bord sous Sources de données. Donnez-lui un nom et elle est prête à recevoir des données.

Les noms suivent quelques règles :

  • Commencer par une lettre minuscule.
  • N'utiliser que des lettres minuscules, des chiffres et des traits de soulignement.
  • Jusqu'à 48 caractères.

Ainsi store_metrics et grid_load_eu sont valides ; Store-Metrics et 2024data ne le sont pas. Une source de données fraîchement créée est vide jusqu'à ce que vous lui envoyiez des observations.

Envoyer des observations

Vous ajoutez des données avec le point de terminaison d'ingestion. Chaque appel transporte le data_source cible et une liste de records. Ici, nous envoyons quelques relevés d'orders dans store_metrics.

curl -X POST https://api.temporis.co/v1/data_sources/ingest \
  -H "Authorization: Bearer ts_<token>" \
  -H "Content-Type: application/json" \
  -d '{
    "data_source": "store_metrics",
    "records": [
      { "timestamp": 1718870400, "name": "orders", "value": 42 },
      { "timestamp": 1718874000, "name": "orders", "value": 51 },
      { "timestamp": 1718877600, "name": "orders", "value": 47 }
    ]
  }'
import requests

requests.post(
    "https://api.temporis.co/v1/data_sources/ingest",
    headers={"Authorization": "Bearer ts_<token>"},
    json={
        "data_source": "store_metrics",
        "records": [
            {"timestamp": 1718870400, "name": "orders", "value": 42},
            {"timestamp": 1718874000, "name": "orders", "value": 51},
            {"timestamp": 1718877600, "name": "orders", "value": 47},
        ],
    },
)  # -> 204 No Content
await fetch("https://api.temporis.co/v1/data_sources/ingest", {
  method: "POST",
  headers: {
    "Authorization": "Bearer ts_<token>",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    data_source: "store_metrics",
    records: [
      { timestamp: 1718870400, name: "orders", value: 42 },
      { timestamp: 1718874000, name: "orders", value: 51 },
      { timestamp: 1718877600, name: "orders", value: 47 },
    ],
  }),
}); // -> 204 No Content

Une ingestion réussie renvoie HTTP 204 No Content avec un corps vide — il n'y a rien à relire, les lignes sont simplement stockées. Consultez la référence d'ingestion pour les détails complets de la requête et des erreurs.

Plusieurs séries dans une seule source

Vous n'êtes pas limité à une série par source. Envoyez différentes valeurs de name dans la même data_source et elles coexistent en tant que séries distinctes. Un magasin pourrait suivre les commandes, le chiffre d'affaires et les visiteurs, tous dans store_metrics.

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718870400, "name": "orders",  "value": 42 },
    { "timestamp": 1718870400, "name": "revenue", "value": 1284.50 }
  ]
}

Garder ensemble des séries connexes est pratique et vous permet de les prévoir conjointement par la suite. Un profil de données est l'endroit où vous choisissez lesquels de ces noms lire réellement.

Mises à jour et remplissages rétroactifs

La clé d'upsert est le couple (timestamp, name). Envoyer un enregistrement dont le timestamp et le nom existent déjà écrase la valeur stockée plutôt que d'ajouter un doublon.

Les remplissages rétroactifs et les nouvelles tentatives sont sans danger

Parce que les écritures sont idempotentes sur (timestamp, name), vous pouvez rejouer un lot après un incident réseau, corriger un relevé erroné ou charger des mois d'historique dans n'importe quel ordre — le résultat est toujours la même valeur finale par seau.

Archivage

Lorsqu'une source ne collecte plus de nouvelles données, vous pouvez l'archiver dans le tableau de bord. L'archivage arrête toute ingestion ultérieure : une requête ciblant une source archivée renvoie HTTP 409 "Data source is archived."

L'archivage est non destructif. Les données existantes restent en place, et tous les profils de données construits sur la source continuent de fonctionner avec les données déjà stockées.
