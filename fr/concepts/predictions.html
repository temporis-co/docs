Prédictions & échantillonnage

Une requête de prédiction renvoie plusieurs futurs possibles plutôt qu'un seul. Cette page explique comment lire la réponse et comment les contrôles d'échantillonnage la façonnent. Pour le schéma exact, consultez la référence predict.

Anatomie d'une requête

Vous établissez une prédiction contre un profil de données, pas contre une source de données brute. Le profil sait déjà quelles séries lire et comment les encoder, donc une requête n'a qu'à indiquer quel profil et comment échantillonner. Quatre champs suffisent :

data_profilestringRequis

Le nom de l'un de vos profils, tel que hourly_orders — ou un profil public.

countintegerRequis

Combien de trajectoires futures indépendantes tirer. Au moins 1.

temperaturenumberRequis

À quel point les échantillons sont dispersés. Doit être supérieur à 0 ; environ 1.0 est neutre.

top_pnumberRequis

Seuil d'échantillonnage par noyau, dans la plage de 0 à 1. Plus bas, il rogne la queue improbable.

Voici une requête pour 20 trajectoires contre le profil hourly_orders :

curl https://api.temporis.co/v1/predict \
  -H "Authorization: Bearer ts_your_token" \
  -H "Content-Type: application/json" \
  -d '{
    "data_profile": "hourly_orders",
    "count": 20,
    "temperature": 1.0,
    "top_p": 0.9
  }'
import requests

resp = requests.post(
    "https://api.temporis.co/v1/predict",
    headers={"Authorization": "Bearer ts_your_token"},
    json={
        "data_profile": "hourly_orders",
        "count": 20,
        "temperature": 1.0,
        "top_p": 0.9,
    },
)
resp.raise_for_status()
result = resp.json()
const resp = await fetch("https://api.temporis.co/v1/predict", {
  method: "POST",
  headers: {
    "Authorization": "Bearer ts_your_token",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    data_profile: "hourly_orders",
    count: 20,
    temperature: 1.0,
    top_p: 0.9,
  }),
});
const result = await resp.json();

La référence complète — chaque champ, code de statut et message d'erreur — se trouve dans la référence du point de terminaison predict.

Lire la réponse

Un appel réussi renvoie HTTP 200 avec quatre champs de premier niveau :

request_idstring

Un identifiant hexadécimal pour cette requête. Conservez-le pour le support et vos propres journaux.

data_profilestring

Reprend le profil que vous avez demandé.

predict_frominteger

Secondes Unix du dernier timestamp observé à partir duquel la prévision se poursuit. Chaque trajectoire commence juste après ce point.

predictionsarray

Une entrée par trajectoire échantillonnée (la longueur égale count). Chacune porte une loss et un tableau data de lignes de prévision.

L'exemple ci-dessous montre deux des trajectoires d'une prévision d'orders. L'interval du profil est 3600 et son patch_spacing est 2, donc les lignes avancent par interval × patch_spacing = 7200 secondes (deux heures) :

{
  "request_id": "9f3c1a7e2b4d",
  "data_profile": "hourly_orders",
  "predict_from": 1718888400,
  "predictions": [
    {
      "loss": 0.84,
      "data": [
        { "timestamp": 1718895600, "orders": 142.0 },
        { "timestamp": 1718902800, "orders": 137.5 }
      ]
    },
    {
      "loss": 1.07,
      "data": [
        { "timestamp": 1718895600, "orders": 151.2 },
        { "timestamp": 1718902800, "orders": 129.8 }
      ]
    }
  ]
}

Chaque ligne possède un timestamp plus un champ par série que le profil lit — ici juste orders. Les timestamps avancent par interval × patch_spacing secondes, le même espacement pour chaque trajectoire, ce qui vous permet d'aligner les échantillons ligne par ligne.

Les contrôles d'échantillonnage

Trois paramètres façonnent l'ensemble de futurs que vous obtenez en retour. Ils arbitrent entre résolution et diversité d'un côté, coût et stabilité de l'autre.

ParamètreCe qu'il faitValeur typiqueAugmentez / diminuez-le quand
countNombre de trajectoires indépendantes tirées. Plus d'échantillons résolvent mieux la distribution.20 à 100Augmentez pour des intervalles plus lisses ; diminuez pour réduire le coût en jetons.
temperatureÀ quel point les échantillons sont dispersés. Plus élevé est plus divers et incertain ; plus bas regroupe près du chemin le plus probable.~1.0Augmentez quand les prévisions semblent trop étroites ; diminuez quand elles se dispersent de manière irréaliste.
top_pSeuil de noyau. Rogne la queue improbable avant l'échantillonnage.0.9 à 1.0Diminuez pour supprimer les sauts surprenants ; augmentez vers 1.0 pour conserver toute la distribution.
Le coût évolue avec count

Chaque échantillon est généré indépendamment et facturé par jeton d'utilisation du modèle, donc une requête avec count 100 coûte environ cinq fois plus qu'une avec count 20. Choisissez count pour équilibrer la résolution dont vous avez besoin et le coût.

Loss : classer les trajectoires

Chaque prédiction porte une loss. Plus elle est faible, plus le modèle a jugé cette trajectoire typique — plus conforme à ce qu'il attendait étant donné l'historique. Les chemins à perte plus élevée restent des échantillons valides ; ils sont simplement moins centraux.

Utilisez loss pour choisir une seule trajectoire représentative, ou comme pondération lorsque vous résumez l'ensemble. Pour saisir le chemin le plus typique :

best = min(result["predictions"], key=lambda p: p["loss"])

Prédictions multivariées

Si un profil lit plusieurs noms de séries, chaque ligne de prévision porte un champ par nom — les séries sont prédites conjointement, de sorte que leurs valeurs restent cohérentes au sein d'une même ligne. Un profil lisant kw et price ensemble renvoie des lignes comme :

{ "timestamp": 1718895600, "kw": 482.0, "price": 31.7 }

Profils de données publics

Vous pouvez établir des prédictions contre des profils partagés que Temporis maintient, sans aucune configuration de votre part — il suffit de passer le nom public comme data_profile :

{ "data_profile": "temporis/polymarket", "count": 20, "temperature": 1.0, "top_p": 0.9 }

La forme de la réponse est identique ; les champs de séries dans chaque ligne correspondent à ce que lit le profil public.
