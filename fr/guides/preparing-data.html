Préparer vos données

De bonnes prévisions commencent par une entrée propre et cohérente. Ce guide explique comment intégrer votre historique dans Temporis sous une forme qu'il peut bien lire.

Utilisez des secondes Unix, de manière cohérente

Chaque timestamp que vous envoyez est un compte entier de secondes Unix — le nombre de secondes depuis le 1970-01-01 00:00:00 UTC. Ni des millisecondes, ni une chaîne de date ISO. Si votre système source vous donne des millisecondes, divisez par 1000 avant l'envoi.

Choisissez une discipline de fuseau horaire et ne vous en écartez jamais. La règle la plus simple qui fonctionne toujours : stockez tout en UTC. Les secondes Unix sont intrinsèquement en UTC, donc tant que vous convertissez chaque timestamp local en son véritable instant UTC avant l'envoi, votre série reste cohérente en interne. Mélanger des valeurs en heure locale et en UTC dans la même série décale silencieusement des points et corrompt le motif quotidien que le modèle apprend.

Vos timestamps n'ont pas besoin de retomber sur des nombres ronds. L'interval d'un profil de données aligne les observations sur une grille régulière — pour le profil hourly_orders, chaque point est regroupé à l'heure la plus proche. Vous fournissez l'instant réel où chaque observation s'est produite ; le profil gère l'alignement.

Une conversion rapide

En Python, int(datetime(2026, 6, 1, tzinfo=timezone.utc).timestamp()) vous donne les secondes Unix pour un instant UTC explicite. Attachez toujours un fuseau horaire pour que la conversion soit sans ambiguïté.

Envoyez l'historique par lots

Vous avez rarement une seule observation à envoyer — vous en avez des mois ou des années. Le point de terminaison d'ingestion accepte de nombreux enregistrements par requête, donc fractionnez votre remplissage rétroactif en lots de quelques milliers d'enregistrements et envoyez-les en boucle plutôt qu'un enregistrement à la fois.

L'ingestion est idempotente. La clé d'upsert est (timestamp, name), donc renvoyer un enregistrement que vous avez déjà envoyé l'écrase simplement avec la même valeur — une opération sans effet. Cela signifie qu'un remplissage rétroactif peut être relancé en toute sécurité : si un lot échoue à mi-chemin ou si votre script plante, relancez-le tout simplement. Vous ne créerez pas de doublons.

import os, requests

token = os.environ["TEMPORIS_TOKEN"]
BASE = "https://api.temporis.co"

# history is a list of (unix_seconds, value) pairs, oldest first
def records_for(history):
    return [{"timestamp": ts, "name": "orders", "value": v} for ts, v in history]

def chunked(seq, size):
    for i in range(0, len(seq), size):
        yield seq[i:i + size]

records = records_for(history)
for batch in chunked(records, 2000):
    resp = requests.post(
        f"{BASE}/v1/data_sources/ingest",
        headers={"Authorization": f"Bearer {token}"},
        json={"data_source": "store_metrics", "records": batch},
    )
    resp.raise_for_status()  # 204 No Content on success
print(f"ingested {len(records)} records")

Une ingestion réussie renvoie HTTP 204 No Content avec un corps vide, donc il n'y a rien à analyser — confirmez simplement le statut. Consultez la référence d'ingestion pour la forme complète de la requête.

Donnez au modèle suffisamment d'historique

Un profil ne peut pas servir de prédictions tant qu'il n'est pas prêt — Temporis a besoin de suffisamment d'historique pour calculer les statistiques de normalisation à travers lesquelles il lit votre série. Et chaque prédiction a besoin d'au moins min_patches de contexte récent, sinon la requête renvoie 409 "Not enough data for this data profile."

La règle pratique : dans le doute, envoyez plus d'historique. Plus d'historique permet à un profil de devenir prêt, dépasse confortablement min_patches et donne au modèle davantage de cycles saisonniers dont s'inspirer. Il n'y a aucune pénalité à envoyer plus que ce que la fenêtre de contexte utilise — les points plus anciens au-delà de num_patches ne sont simplement pas lus au moment de la prédiction.

Pour comprendre ce que signifient min_patches, num_patches et « prêt » et comment ils se rapportent à votre interval, lisez Profils de données.

Timestamps irréguliers et lacunes

Les données réelles sont désordonnées. Les capteurs sautent un relevé, une tâche s'exécute avec quelques minutes de retard, un magasin est fermé un jour férié. Vous n'avez pas besoin de nettoyer tout cela à la main — l'étape de rééchantillonnage du profil s'en charge.

L'interval regroupe vos observations sur une grille régulière, ce qui absorbe les timestamps légèrement irréguliers : un relevé qui arrive à 10:58 et un à 11:03 retombent tous deux dans le seau de 11:00. Pour les seaux manquants, fill_limit définit combien de seaux vides consécutifs Temporis remplira par propagation avant d'abandonner. Une courte lacune est comblée ; une longue lacune est laissée comme un trou et ces lignes sont supprimées plutôt qu'inventées.

Si votre série a des temps morts récurrents — des heures où rien ne peut se produire — ne les traitez pas du tout comme des données manquantes. Un exclude_window masque cette plage horaire afin qu'elle ne déforme jamais la série. C'est un réglage de profil, couvert dans Profils de données.

Vous envoyez les observations brutes telles que vous les avez. Où elles retombent, comment les lacunes sont comblées et quel temps mort est ignoré sont des décisions que prend le profil — pas des transformations que vous intégrez dans les données.

Aligner plusieurs séries

Un profil peut lire plusieurs séries à la fois — un profil multivarié qui les prévoit conjointement. Pour que cela fonctionne bien, les séries ont besoin d'une chronologie commune : envoyez l'observation de chaque série au même ensemble de timestamps afin que les lignes s'alignent après le regroupement.

Vous pouvez envoyer plusieurs séries dans une seule requête d'ingestion — incluez simplement un enregistrement par série à chaque timestamp. Ici, deux séries, orders et visitors, partagent la même chronologie :

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718841600, "name": "orders",   "value": 42 },
    { "timestamp": 1718841600, "name": "visitors", "value": 530 },
    { "timestamp": 1718845200, "name": "orders",   "value": 39 },
    { "timestamp": 1718845200, "name": "visitors", "value": 504 },
    { "timestamp": 1718848800, "name": "orders",   "value": 51 },
    { "timestamp": 1718848800, "name": "visitors", "value": 612 }
  ]
}

Si une série est manquante à un timestamp où une autre est présente, ce seau est rempli selon fill_limit comme n'importe quelle autre lacune — donc quelques retardataires ne casseront pas l'alignement, mais vous obtiendrez la prévision conjointe la plus propre lorsque les séries évoluent sur la même horloge.

Corrections et remplissages rétroactifs

Vous avez trouvé une valeur erronée ? Vous reformulez un trimestre ? Vous ne supprimez ni ne corrigez rien de particulier — vous renvoyez simplement l'enregistrement corrigé. Parce que la clé d'upsert est (timestamp, name), envoyer une nouvelle value à un timestamp+nom existant écrase l'ancienne.

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718841600, "name": "orders", "value": 47 }
  ]
}
Relancer est sans effets de bord

Parce que chaque ingestion est un upsert idempotent, un remplissage rétroactif corrigé peut chevaucher en toute sécurité des données que vous avez déjà envoyées. Renvoyez toute la plage affectée si c'est plus simple que de calculer exactement quels points ont changé.
