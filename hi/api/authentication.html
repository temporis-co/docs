Authentication

प्रत्येक Temporis API request एक bearer token के साथ authenticate होता है। यहाँ बताया गया है कि इसे कैसे बनाएँ, भेजें और सुरक्षित रखें।

Access tokens

आप dashboard में, Access Tokens के अंतर्गत access tokens बनाते हैं। एक token ts_ से उपसर्गित एक लंबी यादृच्छिक string होती है।

पूरा token एक बार दिखाया जाता है, उस क्षण जब आप इसे बनाते हैं। Temporis server-side पर केवल इसका एक hash संग्रहीत करता है, कभी भी token स्वयं नहीं। इसका अर्थ है कि खोया हुआ token पुनर्प्राप्त नहीं किया जा सकता — यदि आप इसे खो देते हैं, तो इसे delete करें और एक नया बनाएँ।

tokens को passwords की तरह मानें

आपके किसी token को रखने वाला कोई भी व्यक्ति data ingest कर सकता है और predictions चला सकता है — और predictions आपके account पर बिल किए जाते हैं। उन्हें तदनुसार सुरक्षित रखें।

किसी request को authenticate करना

token को प्रत्येक request पर Authorization header में Bearer scheme का उपयोग करते हुए भेजें:

Authorization: Bearer ts_your_token_here

इन docs में सभी उदाहरण token को TEMPORIS_TOKEN environment variable से पढ़ते हैं, ताकि secret कभी source में न दिखे। इसे अपने shell में एक बार export करें:

export TEMPORIS_TOKEN="ts_your_token_here"

फिर इसे एक header के रूप में संलग्न करें:

curl https://api.temporis.co/v1/predict \
  -H "Authorization: Bearer $TEMPORIS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{ "data_profile": "hourly_orders", "count": 5, "temperature": 1.0, "top_p": 0.9 }'
import os, requests

token = os.environ["TEMPORIS_TOKEN"]

resp = requests.post(
    "https://api.temporis.co/v1/predict",
    headers={"Authorization": f"Bearer {token}"},
    json={"data_profile": "hourly_orders", "count": 5, "temperature": 1.0, "top_p": 0.9},
)
resp.raise_for_status()
const token = process.env.TEMPORIS_TOKEN;

const resp = await fetch("https://api.temporis.co/v1/predict", {
  method: "POST",
  headers: {
    "Authorization": `Bearer ${token}`,
    "Content-Type": "application/json",
  },
  body: JSON.stringify({ data_profile: "hourly_orders", count: 5, temperature: 1.0, top_p: 0.9 }),
});

जब authentication विफल हो

यदि Authorization header अनुपस्थित या विकृत है, या token अज्ञात है, तो API HTTP 401 और एक JSON body के साथ प्रतिक्रिया देता है जो बताता है कि क्या गलत हुआ:

{ "detail": "Invalid access token." }

अनुपस्थित या विकृत header इसके बजाय { "detail": "Missing access token." } लौटाता है। दोनों 401 हैं। प्रतिक्रियाओं की पूरी सूची के लिए Errors और status codes देखें।

tokens को सुरक्षित रखना

कुछ आदतें आपके tokens को गलत हाथों से दूर रखती हैं:

  • tokens को कभी source control में commit न करें, और उन्हें कभी client-side code में न रखें। Browsers और mobile apps अपना source उपयोगकर्ताओं तक भेजते हैं।
  • tokens को environment variables या किसी secrets manager से पढ़ें। उन्हें logs, screenshots, और tickets से दूर रखें।
  • प्रति service एक अलग token बनाएँ ताकि आप दूसरों को बाधित किए बिना किसी एक को revoke कर सकें।
  • token को तुरंत revoke करने के लिए उसे dashboard में delete करें। Deletion तुरंत प्रभावी होता है — उस token का उपयोग करने वाली अगली request को 401 मिलता है।
