Errors & status codes

Temporis सफलता और विफलता की रिपोर्ट कैसे करता है, प्रत्येक status का क्या अर्थ है, और इन्हें कैसे संभालें।

Response आकार

किसी response का आकार इस पर निर्भर करता है कि क्या हुआ:

  • एक सफल ingest बिना किसी body के 204 No Content लौटाता है।
  • एक सफल prediction एक JSON body के साथ 200 लौटाता है।
  • कोई भी error संबंधित HTTP status को { "detail": "..." } रूप की एक JSON body के साथ लौटाता है।
{ "detail": "Data profile not found." }

Errors को संभालना

अधिकांश विफलताएं एक स्पष्ट अगली कार्रवाई से मेल खाती हैं:

  • 401 — जांचें कि Authorization header मौजूद है और token वैध है। Authentication देखें।
  • 402 — dashboard में Billing के अंतर्गत credit card जोड़ें या ठीक करें, फिर दोबारा प्रयास करें।
  • 404 — सुनिश्चित करें कि data source या data profile मौजूद है, और profile serve के लिए तैयार है।
  • predict पर 409 — तब तक और history ingest करें जब तक profile अपने न्यूनतम तक न पहुंचे, फिर दोबारा प्रयास करें।
  • ingest पर 409 — data source archived है; इसे unarchive करें या किसी अन्य source में लिखें।
  • 502 — क्षणिक। exponential backoff के साथ दोबारा प्रयास करें।

status code पर एक सरल branch इन सभी को कवर कर लेता है:

import time, requests

def predict(payload, token, retries=3):
    for attempt in range(retries):
        resp = requests.post(
            "https://api.temporis.co/v1/predict",
            headers={"Authorization": f"Bearer {token}"},
            json=payload,
        )
        if resp.status_code == 200:
            return resp.json()
        if resp.status_code == 502:
            time.sleep(2 ** attempt)   # transient: back off and retry
            continue
        detail = resp.json().get("detail", "")
        if resp.status_code == 402:
            raise RuntimeError(f"Billing: {detail} — add a card in the dashboard.")
        if resp.status_code == 409:
            raise RuntimeError(f"Not ready: {detail} — ingest more history.")
        resp.raise_for_status()        # 400 / 401 / 404: fix the request
    raise RuntimeError("Prediction failed after retries.")

Retries

सही चीज़ों को retry करें

502 को retry करना सुरक्षित है — यह क्षणिक है। Ingest idempotent है (writes (timestamp, name) key पर upsert होते हैं), इसलिए किसी विफल ingest को retry करना हमेशा सुरक्षित है। 4xx responses को आंख मूंदकर retry करने से बचें; पहले request को ठीक करें।
