Data sources

data source वह जगह है जहाँ आपके raw observations रहते हैं। यह पेज Temporis के data model को और इसमें डेटा अच्छी तरह कैसे भेजें, इसे कवर करता है।

आप dashboard में एक बार data source बनाते हैं, फिर जैसे-जैसे observations होते हैं, उन्हें API के ज़रिए इसमें stream करते रहते हैं।

data model

Temporis डेटा को rows के रूप में संग्रहित करता है। हर observation एक अकेला triple होता है: कब कुछ मापा गया, क्या मापा गया, और जो संख्या आपने मापी।

timestamp
Unix time सेकंड्स में (एक integer)। वह क्षण जब observation लिया गया।
name
वह series जिससे यह value संबंधित है — उदाहरण के लिए orders या revenue।
value
एक संख्या। Floats भी ठीक हैं।

एक अकेला data source कई named series को एक साथ रख सकता है। name field ही उन्हें अलग रखता है। आपका डेटा हर दूसरे ग्राहक के डेटा से अलग रहता है — एक data source सिर्फ़ आपका होता है।

data source बनाना

आप dashboard में Data Sources के अंतर्गत data source बनाते हैं। इसे एक नाम दें और यह डेटा प्राप्त करने के लिए तैयार है।

नाम कुछ नियमों का पालन करते हैं:

  • एक lowercase अक्षर से शुरू करें।
  • केवल lowercase अक्षर, अंक और underscores का उपयोग करें।
  • अधिकतम 48 अक्षर तक।

इसलिए store_metrics और grid_load_eu मान्य हैं; Store-Metrics और 2024data नहीं हैं। नया बनाया गया data source तब तक खाली रहता है जब तक आप इसे observations नहीं भेजते।

observations भेजना

आप ingest endpoint से डेटा जोड़ते हैं। हर call में target data_source और records की एक list होती है। यहाँ हम store_metrics में कुछ orders readings push कर रहे हैं।

curl -X POST https://api.temporis.co/v1/data_sources/ingest \
  -H "Authorization: Bearer ts_<token>" \
  -H "Content-Type: application/json" \
  -d '{
    "data_source": "store_metrics",
    "records": [
      { "timestamp": 1718870400, "name": "orders", "value": 42 },
      { "timestamp": 1718874000, "name": "orders", "value": 51 },
      { "timestamp": 1718877600, "name": "orders", "value": 47 }
    ]
  }'
import requests

requests.post(
    "https://api.temporis.co/v1/data_sources/ingest",
    headers={"Authorization": "Bearer ts_<token>"},
    json={
        "data_source": "store_metrics",
        "records": [
            {"timestamp": 1718870400, "name": "orders", "value": 42},
            {"timestamp": 1718874000, "name": "orders", "value": 51},
            {"timestamp": 1718877600, "name": "orders", "value": 47},
        ],
    },
)  # -> 204 No Content
await fetch("https://api.temporis.co/v1/data_sources/ingest", {
  method: "POST",
  headers: {
    "Authorization": "Bearer ts_<token>",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    data_source: "store_metrics",
    records: [
      { timestamp: 1718870400, name: "orders", value: 42 },
      { timestamp: 1718874000, name: "orders", value: 51 },
      { timestamp: 1718877600, name: "orders", value: 47 },
    ],
  }),
}); // -> 204 No Content

एक सफल ingest HTTP 204 No Content लौटाता है जिसका body खाली होता है — पढ़ने के लिए कुछ नहीं होता, rows बस संग्रहित हो जाती हैं। पूरे request और error विवरण के लिए ingest reference देखें।

एक source में कई series

आप प्रति source एक series तक सीमित नहीं हैं। एक ही data_source में अलग-अलग name values भेजें और वे अलग series के रूप में साथ-साथ रहती हैं। एक store, store_metrics में orders, revenue और visitors सभी को track कर सकता है।

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718870400, "name": "orders",  "value": 42 },
    { "timestamp": 1718870400, "name": "revenue", "value": 1284.50 }
  ]
}

संबंधित series को एक साथ रखना सुविधाजनक है और बाद में आपको उन्हें संयुक्त रूप से forecast करने देता है। data profile वह जगह है जहाँ आप चुनते हैं कि इन नामों में से किसे असल में पढ़ना है।

Updates और backfills

upsert key (timestamp, name) की जोड़ी है। ऐसा record भेजना जिसका timestamp और name पहले से मौजूद हैं, संग्रहित value को duplicate जोड़ने के बजाय overwrite कर देता है।

Backfills और retries सुरक्षित हैं

चूँकि writes (timestamp, name) पर idempotent हैं, आप network में रुकावट के बाद एक batch दोबारा भेज सकते हैं, गलत reading सुधार सकते हैं, या महीनों का इतिहास किसी भी क्रम में load कर सकते हैं — प्रति bucket अंतिम value हमेशा वही रहेगी।

Archiving

जब कोई source नया डेटा एकत्र नहीं कर रहा हो, तो आप उसे dashboard में archive कर सकते हैं। Archiving आगे की ingestion रोक देती है: किसी archived source को target करने वाला request HTTP 409 "Data source is archived." लौटाता है।

Archiving non-destructive है। मौजूदा डेटा अपनी जगह बना रहता है, और source पर बने कोई भी data profiles पहले से संग्रहित डेटा के विरुद्ध काम करते रहते हैं।
