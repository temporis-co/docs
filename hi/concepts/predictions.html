Predictions और sampling

एक prediction request एक के बजाय कई संभावित भविष्य लौटाता है। यह पृष्ठ बताता है कि response को कैसे पढ़ें और sampling controls इसे कैसे आकार देते हैं। सटीक schema के लिए predict reference देखें।

एक request की संरचना

आप किसी data profile के विरुद्ध prediction करते हैं, raw data source के विरुद्ध नहीं। profile पहले से जानता है कि कौन-सी series पढ़नी हैं और उन्हें कैसे encode करना है, इसलिए एक request को केवल यह बताना होता है कि कौन-सा profile और कैसे sample करना है। चार fields इसे कवर कर देते हैं:

data_profilestringआवश्यक

आपके किसी profile का नाम, जैसे hourly_orders — या कोई public profile।

countintegerआवश्यक

कितनी स्वतंत्र भविष्य की trajectories निकालनी हैं। कम से कम 1।

temperaturenumberआवश्यक

samples कितने फैले हुए हैं। 0 से अधिक होना चाहिए; लगभग 1.0 तटस्थ होता है।

top_pnumberआवश्यक

0 से 1 की सीमा में nucleus-sampling cutoff। कम मान असंभावित tail को काट देता है।

यहाँ hourly_orders profile के विरुद्ध 20 trajectories के लिए एक request है:

curl https://api.temporis.co/v1/predict \
  -H "Authorization: Bearer ts_your_token" \
  -H "Content-Type: application/json" \
  -d '{
    "data_profile": "hourly_orders",
    "count": 20,
    "temperature": 1.0,
    "top_p": 0.9
  }'
import requests

resp = requests.post(
    "https://api.temporis.co/v1/predict",
    headers={"Authorization": "Bearer ts_your_token"},
    json={
        "data_profile": "hourly_orders",
        "count": 20,
        "temperature": 1.0,
        "top_p": 0.9,
    },
)
resp.raise_for_status()
result = resp.json()
const resp = await fetch("https://api.temporis.co/v1/predict", {
  method: "POST",
  headers: {
    "Authorization": "Bearer ts_your_token",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    data_profile: "hourly_orders",
    count: 20,
    temperature: 1.0,
    top_p: 0.9,
  }),
});
const result = await resp.json();

पूरा reference — हर field, status code, और error message — predict endpoint reference में है।

Response को पढ़ना

एक सफल call HTTP 200 के साथ चार top-level fields लौटाता है:

request_idstring

इस request के लिए एक hex identifier। इसे support और अपने स्वयं के logs के लिए संभालकर रखें।

data_profilestring

आपके द्वारा माँगे गए profile को प्रतिध्वनित करता है।

predict_frominteger

अंतिम observed timestamp का Unix seconds, जहाँ से forecast जारी रहता है। हर trajectory इस बिंदु के ठीक बाद से शुरू होती है।

predictionsarray

प्रत्येक sampled trajectory के लिए एक entry (लंबाई count के बराबर)। प्रत्येक में एक loss और forecast rows का एक data array होता है।

नीचे दिया गया उदाहरण किसी orders forecast से दो trajectories दिखाता है। profile का interval 3600 है और इसका patch_spacing 2 है, इसलिए rows interval × patch_spacing = 7200 seconds (दो घंटे) के अंतराल पर बढ़ती हैं:

{
  "request_id": "9f3c1a7e2b4d",
  "data_profile": "hourly_orders",
  "predict_from": 1718888400,
  "predictions": [
    {
      "loss": 0.84,
      "data": [
        { "timestamp": 1718895600, "orders": 142.0 },
        { "timestamp": 1718902800, "orders": 137.5 }
      ]
    },
    {
      "loss": 1.07,
      "data": [
        { "timestamp": 1718895600, "orders": 151.2 },
        { "timestamp": 1718902800, "orders": 129.8 }
      ]
    }
  ]
}

प्रत्येक row में एक timestamp होता है साथ ही profile द्वारा पढ़ी जाने वाली प्रत्येक series के लिए एक field — यहाँ केवल orders। timestamps interval × patch_spacing seconds के अंतराल पर बढ़ती हैं, हर trajectory के लिए समान spacing, ताकि आप samples को row दर row align कर सकें।

sampling controls

तीन parameters उन भविष्यों के समूह को आकार देते हैं जो आपको वापस मिलते हैं। ये resolution और diversity को cost तथा stability के विरुद्ध संतुलित करते हैं।

Parameterयह क्या करता हैसामान्य मानइसे कब बढ़ाएँ / घटाएँ
countनिकाली गई स्वतंत्र trajectories की संख्या। अधिक samples distribution को बेहतर ढंग से resolve करते हैं।20 से 100अधिक smooth intervals के लिए बढ़ाएँ; token cost घटाने के लिए कम करें।
temperaturesamples कितने फैले हुए हैं। अधिक मान अधिक विविध और अनिश्चित होता है; कम मान सबसे संभावित पथ के निकट गुच्छित करता है।~1.0जब forecasts बहुत संकीर्ण दिखें तब बढ़ाएँ; जब वे अवास्तविक रूप से बिखरें तब कम करें।
top_pNucleus cutoff। sampling से पहले असंभावित tail को काट देता है।0.9 से 1.0आश्चर्यजनक उछाल दबाने के लिए कम करें; पूरे distribution को बनाए रखने के लिए 1.0 की ओर बढ़ाएँ।
cost, count के साथ बढ़ता है

प्रत्येक sample स्वतंत्र रूप से उत्पन्न होता है और model उपयोग के प्रति token पर मीटर किया जाता है, इसलिए count 100 वाली request की लागत count 20 वाली request से लगभग पाँच गुना होती है। अपनी आवश्यक resolution को cost के विरुद्ध संतुलित करने के लिए count चुनें।

Loss: trajectories को क्रमबद्ध करना

प्रत्येक prediction में एक loss होता है। कम मान का अर्थ है कि model ने उस trajectory को अधिक सामान्य माना — history को देखते हुए जो उसने अपेक्षित किया था उसके अधिक अनुरूप। अधिक-loss वाले पथ अब भी वैध samples हैं; वे बस कम केंद्रीय होते हैं।

किसी एकल प्रतिनिधि trajectory को चुनने के लिए loss का उपयोग करें, या जब आप समूह का सारांश बनाते हैं तब एक weight के रूप में। सबसे सामान्य पथ पकड़ने के लिए:

best = min(result["predictions"], key=lambda p: p["loss"])

Multivariate predictions

यदि कोई profile कई series names पढ़ता है, तो प्रत्येक forecast row में प्रत्येक name के लिए एक field होता है — series का संयुक्त रूप से prediction किया जाता है, इसलिए उनके मान एक row के भीतर सुसंगत रहते हैं। kw और price को एक साथ पढ़ने वाला profile इस तरह की rows लौटाता है:

{ "timestamp": 1718895600, "kw": 482.0, "price": 31.7 }

Public data profiles

आप Temporis द्वारा बनाए गए साझा profiles के विरुद्ध prediction कर सकते हैं, बिना अपनी ओर से किसी setup के — बस public name को data_profile के रूप में पास करें:

{ "data_profile": "temporis/polymarket", "count": 20, "temperature": 1.0, "top_p": 0.9 }

response की संरचना समान होती है; प्रत्येक row में series fields उसी से मेल खाते हैं जो public profile पढ़ता है।
