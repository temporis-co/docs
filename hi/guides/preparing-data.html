अपना डेटा तैयार करना

अच्छे पूर्वानुमान साफ, सुसंगत इनपुट से शुरू होते हैं। यह गाइड बताती है कि अपने इतिहास को एक ऐसे आकार में Temporis में कैसे लाएँ जिसे वह अच्छी तरह पढ़ सके।

लगातार Unix seconds का इस्तेमाल करें

आप जो भी timestamp भेजते हैं वह Unix seconds की एक पूर्णांक गिनती होती है — 1970-01-01 00:00:00 UTC के बाद बीते सेकंडों की संख्या। मिलीसेकंड नहीं, और न ही कोई ISO date string। अगर आपका स्रोत सिस्टम आपको मिलीसेकंड देता है, तो भेजने से पहले 1000 से भाग दें।

एक timezone अनुशासन चुनें और उससे कभी विचलित न हों। सबसे सरल नियम जो हमेशा काम करता है: सब कुछ UTC में संग्रहित करें। Unix seconds स्वभाव से UTC होते हैं, इसलिए जब तक आप भेजने से पहले हर स्थानीय timestamp को उसके असली UTC क्षण में बदलते हैं, आपकी series आंतरिक रूप से सुसंगत रहती है। एक ही series में local-time और UTC मानों को मिलाने से बिंदु चुपचाप इधर-उधर खिसक जाते हैं और मॉडल जो दैनिक पैटर्न सीखता है वह बिगड़ जाता है।

आपके timestamps को गोल संख्याओं पर आना ज़रूरी नहीं है। एक data profile का interval अवलोकनों को एक नियमित ग्रिड पर स्नैप कर देता है — hourly_orders profile के लिए हर बिंदु को निकटतम घंटे में बकेट कर दिया जाता है। आप वह असली क्षण देते हैं जब प्रत्येक अवलोकन हुआ; संरेखण को profile संभालती है।

एक त्वरित रूपांतरण

Python में, int(datetime(2026, 6, 1, tzinfo=timezone.utc).timestamp()) आपको एक स्पष्ट UTC क्षण के लिए Unix seconds देता है। हमेशा एक timezone जोड़ें ताकि रूपांतरण निःसंदेह हो।

इतिहास को बैचों में भेजें

आपके पास भेजने के लिए शायद ही कभी एक अवलोकन होता है — आपके पास महीनों या वर्षों का होता है। ingest endpoint प्रति अनुरोध कई records लेता है, इसलिए अपने backfill को कुछ हज़ार records के बैचों में बाँटें और उन्हें एक-एक record के बजाय एक लूप में पोस्ट करें।

Ingestion idempotent है। upsert कुंजी (timestamp, name) है, इसलिए जो record आप पहले ही भेज चुके हैं उसे फिर से भेजना उसे उसी मान से अधिलेखित कर देता है — एक no-op। इसका मतलब है कि एक backfill को फिर से चलाना सुरक्षित है: अगर कोई बैच बीच में विफल हो जाता है या आपकी script क्रैश हो जाती है, तो बस उसे फिर से शुरू करें। आप डुप्लिकेट नहीं बनाएँगे।

import os, requests

token = os.environ["TEMPORIS_TOKEN"]
BASE = "https://api.temporis.co"

# history is a list of (unix_seconds, value) pairs, oldest first
def records_for(history):
    return [{"timestamp": ts, "name": "orders", "value": v} for ts, v in history]

def chunked(seq, size):
    for i in range(0, len(seq), size):
        yield seq[i:i + size]

records = records_for(history)
for batch in chunked(records, 2000):
    resp = requests.post(
        f"{BASE}/v1/data_sources/ingest",
        headers={"Authorization": f"Bearer {token}"},
        json={"data_source": "store_metrics", "records": batch},
    )
    resp.raise_for_status()  # 204 No Content on success
print(f"ingested {len(records)} records")

एक सफल ingest HTTP 204 No Content के साथ एक खाली body लौटाता है, इसलिए पार्स करने के लिए कुछ नहीं होता — बस status की पुष्टि करें। पूरे अनुरोध के आकार के लिए ingest reference देखें।

मॉडल को पर्याप्त इतिहास दें

एक profile तब तक पूर्वानुमान नहीं दे सकती जब तक वह ready न हो जाए — Temporis को normalization आँकड़ों की गणना करने के लिए पर्याप्त इतिहास चाहिए जिनके माध्यम से वह आपकी series पढ़ती है। और हर पूर्वानुमान को कम से कम min_patches जितने हाल के संदर्भ की ज़रूरत होती है, वरना अनुरोध 409 "Not enough data for this data profile." लौटाता है।

व्यावहारिक नियम: संदेह हो तो ज़्यादा इतिहास भेजें। ज़्यादा इतिहास एक profile को ready बनने देता है, min_patches को आराम से पार करता है, और मॉडल को सीखने के लिए ज़्यादा मौसमी चक्र देता है। context window जितने का उपयोग करता है उससे अधिक भेजने का कोई दंड नहीं है — num_patches से परे पुराने बिंदु predict समय पर बस नहीं पढ़े जाते।

यह जानने के लिए कि min_patches, num_patches, और "ready" का क्या अर्थ है और वे आपके interval से कैसे संबंधित हैं, Data profiles पढ़ें।

अनियमित timestamps और अंतराल

असली डेटा गंदा होता है। सेंसर एक रीडिंग छोड़ देते हैं, कोई job कुछ मिनट देर से चलती है, छुट्टी के दिन एक स्टोर बंद रहता है। आपको इनमें से किसी को हाथ से साफ करने की ज़रूरत नहीं है — profile का resampling चरण इसे संभालता है।

interval आपके अवलोकनों को एक नियमित ग्रिड पर बकेट करता है, जो थोड़े अनियमित timestamps को अवशोषित कर लेता है: 10:58 पर आने वाली एक रीडिंग और 11:03 पर आने वाली दोनों 11:00 के बकेट में आ जाती हैं। गायब बकेटों के लिए, fill_limit तय करता है कि हार मानने से पहले Temporis कितने लगातार खाली बकेटों को forward-fill करेगा। एक छोटा अंतराल पाट दिया जाता है; एक लंबे अंतराल को एक छेद के रूप में छोड़ दिया जाता है और उन पंक्तियों को गढ़ने के बजाय हटा दिया जाता है।

अगर आपकी series में आवर्ती मृत समय है — ऐसे घंटे जब कुछ भी संभवतः नहीं हो सकता — तो इसे गायब डेटा मानें ही नहीं। एक exclude_window उस दिन-के-समय की सीमा को मास्क कर देता है ताकि वह series को कभी विकृत न करे। यह एक profile सेटिंग है, जिसे Data profiles में समझाया गया है।

आप कच्चे अवलोकन वैसे ही भेजते हैं जैसे आपके पास हैं। वे कहाँ आते हैं, अंतराल कैसे भरे जाते हैं, और किस मृत समय को नज़रअंदाज़ किया जाता है — ये निर्णय profile करती है, न कि वे रूपांतरण जो आप डेटा में पकाते हैं।

कई series को संरेखित करना

एक profile एक साथ कई series पढ़ सकती है — एक multivariate profile जो उनका संयुक्त रूप से पूर्वानुमान लगाती है। इसके अच्छे से काम करने के लिए, series को एक साझा timeline चाहिए: प्रत्येक series का अवलोकन समान timestamps के सेट पर भेजें ताकि bucketing के बाद पंक्तियाँ मेल खाएँ।

आप एक ही ingest अनुरोध में कई series भेज सकते हैं — बस प्रत्येक timestamp पर प्रति series एक record शामिल करें। यहाँ दो series, orders और visitors, एक ही timeline साझा करती हैं:

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718841600, "name": "orders",   "value": 42 },
    { "timestamp": 1718841600, "name": "visitors", "value": 530 },
    { "timestamp": 1718845200, "name": "orders",   "value": 39 },
    { "timestamp": 1718845200, "name": "visitors", "value": 504 },
    { "timestamp": 1718848800, "name": "orders",   "value": 51 },
    { "timestamp": 1718848800, "name": "visitors", "value": 612 }
  ]
}

अगर एक series ऐसे timestamp पर गायब है जहाँ दूसरी मौजूद है, तो उस बकेट को किसी भी अन्य अंतराल की तरह fill_limit के अनुसार भरा जाता है — इसलिए कुछ छिटपुट बिंदु संरेखण को नहीं तोड़ेंगे, पर सबसे साफ संयुक्त पूर्वानुमान तब मिलेगा जब series एक ही घड़ी पर चलें।

सुधार और backfills

कोई गलत मान मिला? किसी तिमाही को फिर से बता रहे हैं? आप कुछ खास हटाते या पैच नहीं करते — आप बस सुधारा हुआ record फिर से भेजते हैं। चूँकि upsert कुंजी (timestamp, name) है, किसी मौजूदा timestamp+name पर एक नया value भेजना पुराने को अधिलेखित कर देता है।

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718841600, "name": "orders", "value": 47 }
  ]
}
फिर से चलाना दुष्प्रभावों से मुक्त है

चूँकि हर ingest एक idempotent upsert है, एक सुधारा हुआ backfill उस डेटा के साथ सुरक्षित रूप से ओवरलैप कर सकता है जो आप पहले ही भेज चुके हैं। अगर यह गणना करने से आसान हो कि कौन से बिंदु बदले, तो पूरी प्रभावित सीमा को फिर से पोस्ट करें।
