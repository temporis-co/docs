क्विकस्टार्ट

यह गाइड आपको एक नए अकाउंट से आपके पहले forecast तक ले जाती है। आप एक API key बनाएँगे, एक data source register करेंगे, कुछ history stream करेंगे, एक data profile बनाएँगे, और prediction endpoint को call करेंगे। इसके लिए लगभग पाँच मिनट रखें।

दो surfaces, एक platform. आप चीज़ें dashboard में सेट करेंगे (keys, data sources, data profiles, billing) और data व forecasting का काम API के ज़रिए करेंगे। यह पेज दोनों के बारे में बताता है।

  1. एक access token बनाएँ

    dashboard में, Access Tokens खोलें और एक बनाएँ। token एक बार ही दिखाया जाता है — इसे तुरंत copy कर लें। यह ts_… जैसा दिखता है और हर API request को authenticate करता है।

    इसे एक environment variable के रूप में store करें ताकि नीचे दिए गए examples वैसे के वैसे काम करें:

    export TEMPORIS_TOKEN="ts_your_token_here"

  2. एक payment method जोड़ें

    Prediction usage के हर token के आधार पर metered होता है, इसलिए /v1/predict चलने से पहले एक card file पर होना आवश्यक है। इसे Billing के अंतर्गत जोड़ें। Data ingest करने के लिए billing की ज़रूरत नहीं है — केवल forecasting के लिए होती है।

  3. एक data source बनाएँ

    Data Sources के अंतर्गत, store_metrics नाम का एक बनाएँ। नाम एक lowercase अक्षर से शुरू होने चाहिए और इनमें केवल lowercase अक्षर, digits, और underscores हों। जब तक आप उसे data नहीं भेजते, तब तक एक data source एक खाली container होता है।

  4. कुछ history भेजें

    हर record एक (timestamp, name, value) triple होता है। timestamp seconds में एक Unix time है, name series है, और value संख्या है। यहाँ हम कुछ hourly orders readings भेजते हैं; व्यवहार में आप जितनी real history आपके पास हो उतनी भेजेंगे।

  5. एक data profile बनाएँ

    dashboard में वापस जाकर, Data Profiles खोलें और hourly_orders नाम का एक बनाएँ। इसे store_metrics data source की ओर इंगित करें, orders series चुनें, और interval को 3600 seconds (एक घंटा) पर सेट करें। बाकी सब चीज़ों के लिए defaults एक अच्छा शुरुआती बिंदु हैं।

    Temporis द्वारा किसी profile को serving के लिए तैयार करने से पहले उसे history की एक baseline चाहिए। जब उसके पास पर्याप्त data हो जाता है, तो dashboard उसे ready के रूप में चिह्नित कर देता है। तब तक, predictions एक "not ready" error लौटाते हैं — यह अपेक्षित है।

  6. एक forecast generate करें

    अब profile से पूछें कि आगे क्या आता है। count यह है कि कितनी स्वतंत्र भविष्य की trajectories निकालनी हैं; temperature और top_p यह नियंत्रित करते हैं कि sampling कितनी साहसिक है।

अभी क्या हुआ

आपने observations को एक data source में stream किया, उन्हें कैसे पढ़ना है यह एक data profile के साथ बताया, और /v1/predict से पाँच संभावित भविष्य निकाले। हर prediction एक पूर्ण trajectory है; मिलकर ये आगे क्या आता है इसकी एक probabilistic तस्वीर बनाती हैं। किसी prediction का loss जितना कम होगा, model ने उस trajectory को उतना ही typical माना।
