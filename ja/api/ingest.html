レコードの取り込み

観測値をデータソースへ送信します。データソースはあらかじめ存在し(ダッシュボードで作成)、アーカイブされていない必要があります。

POST/v1/data_sources/ingest

リクエストボディ

ボディには、対象のデータソース名と、レコードのバッチを指定します。

data_sourcestring必須

書き込み先となる、既存かつアーカイブされていないデータソースの名前。

recordsarray必須

保存するレコードのリスト。各レコードは、1 つの系列のある時点における 1 つの観測値です。

レコードのフィールド

timestampinteger必須

秒単位の Unix 時刻。

namestring必須

系列名。1 つのデータソースには、名前付きの系列を複数保持できます。

valuenumber必須

観測された値(整数または浮動小数点数)。

リクエスト例

以下は、環境変数からトークンを読み取り、4 件の orders の観測値を store_metrics データソースに書き込む例です:

curl https://api.temporis.co/v1/data_sources/ingest \
  -H "Authorization: Bearer $TEMPORIS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "data_source": "store_metrics",
    "records": [
      { "timestamp": 1718841600, "name": "orders", "value": 42 },
      { "timestamp": 1718845200, "name": "orders", "value": 51 },
      { "timestamp": 1718848800, "name": "orders", "value": 47 },
      { "timestamp": 1718852400, "name": "orders", "value": 63 }
    ]
  }'

レスポンス

成功時、このエンドポイントは空のボディとともに 204 No Content を返します:

204 No Content

アップサートの動作

各レコードは (timestamp, name) のペアをキーとします。同じタイムスタンプと系列名を持つレコードを再送すると、保存済みの値が上書きされます。

リトライとバックフィルは冪等です

書き込みはアップサートであるため、失敗したリクエストの再試行や、過去データの再送を安全に行えます。重複は蓄積されず、同じキーにまとまります。

バッチ処理

1 リクエストあたり 1 レコードを送るよりも、多数のレコードを 1 つのリクエストで送ることをお勧めします。バッチ処理により、ラウンドトリップとオーバーヘッドが大幅に削減されます。履歴を読み込む際は、1 ポイントずつストリーミングするのではなく、多数のレコードからなるチャンク単位で送信してください。

エラー

エラーは、HTTP ステータスと、{ "detail": "..." } という形式の JSON ボディを返します。

ステータスdetail メッセージ意味
401Invalid access token. / Missing access token.認証に失敗しました。トークンを確認してください。
404Data source not found.指定された名前のデータソースが存在しません。先にダッシュボードで作成してください。
409Data source is archived.データソースがアーカイブされており、書き込みを受け付けられません。アーカイブを解除するか、別のソースを使用してください。

完全なリファレンスについては、エラーとステータスコードを参照してください。
