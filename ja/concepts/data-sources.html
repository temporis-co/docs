データソース

データソースは、生の観測値が格納される場所です。このページでは、Temporis のデータモデルと、そこへ適切にデータを送る方法を説明します。

データソースはダッシュボードで一度だけ作成し、その後は観測値が発生するたびに API 経由でストリーミングしていきます。

データモデル

Temporis はデータを行として保存します。すべての観測値は、ひとつの三つ組です。すなわち、何かが測定されたとき、何が測定されたか、そして測定した数値です。

timestamp
秒単位の Unix 時刻（整数）。観測が取得された瞬間です。
name
この値が属する系列です。たとえば orders や revenue などです。
value
数値です。浮動小数点数でも問題ありません。

1 つのデータソースは、名前付きの系列を多数並べて保持できます。それらを区別するのが name フィールドです。あなたのデータは他のすべての顧客のデータから隔離されています。データソースはあなただけのものです。

データソースの作成

データソースはダッシュボードのデータソースセクションで作成します。名前を付ければ、データを受け取る準備が整います。

名前にはいくつかのルールがあります。

  • 小文字で始めること。
  • 小文字、数字、アンダースコアのみを使うこと。
  • 最大 48 文字まで。

したがって store_metrics や grid_load_eu は有効ですが、Store-Metrics や 2024data は無効です。作成したばかりのデータソースは、観測値を送るまで空のままです。

観測値の送信

データの追加には ingest エンドポイントを使います。各呼び出しには、対象の data_source と records のリストを含めます。ここでは、いくつかの orders の値を store_metrics に投入します。

curl -X POST https://api.temporis.co/v1/data_sources/ingest \
  -H "Authorization: Bearer ts_<token>" \
  -H "Content-Type: application/json" \
  -d '{
    "data_source": "store_metrics",
    "records": [
      { "timestamp": 1718870400, "name": "orders", "value": 42 },
      { "timestamp": 1718874000, "name": "orders", "value": 51 },
      { "timestamp": 1718877600, "name": "orders", "value": 47 }
    ]
  }'
import requests

requests.post(
    "https://api.temporis.co/v1/data_sources/ingest",
    headers={"Authorization": "Bearer ts_<token>"},
    json={
        "data_source": "store_metrics",
        "records": [
            {"timestamp": 1718870400, "name": "orders", "value": 42},
            {"timestamp": 1718874000, "name": "orders", "value": 51},
            {"timestamp": 1718877600, "name": "orders", "value": 47},
        ],
    },
)  # -> 204 No Content
await fetch("https://api.temporis.co/v1/data_sources/ingest", {
  method: "POST",
  headers: {
    "Authorization": "Bearer ts_<token>",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    data_source: "store_metrics",
    records: [
      { timestamp: 1718870400, name: "orders", value: 42 },
      { timestamp: 1718874000, name: "orders", value: 51 },
      { timestamp: 1718877600, name: "orders", value: 47 },
    ],
  }),
}); // -> 204 No Content

取り込みに成功すると、空のボディを伴う HTTP 204 No Content が返されます。読み戻すものは何もなく、行が単に保存されるだけです。完全なリクエストとエラーの詳細についてはingest リファレンスを参照してください。

1 つのソースに複数の系列

1 つのソースにつき 1 つの系列という制限はありません。異なる name の値を同じ data_source に送れば、それらは別々の系列として共存します。ある店舗では、orders、revenue、visitors をすべて store_metrics で追跡することもできるでしょう。

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718870400, "name": "orders",  "value": 42 },
    { "timestamp": 1718870400, "name": "revenue", "value": 1284.50 }
  ]
}

関連する系列をまとめておくと便利であり、後でそれらを同時に予測できるようにもなります。これらの名前のうち、実際にどれを読み取るかを選ぶのがデータプロファイルです。

更新とバックフィル

アップサートのキーは (timestamp, name) のペアです。timestamp と name がすでに存在するレコードを送ると、重複を追加するのではなく、保存されている値が上書きされます。

バックフィルとリトライは安全です

書き込みは (timestamp, name) に対して冪等なので、ネットワークの不調の後にバッチを再送したり、誤った値を修正したり、数か月分の履歴を任意の順序で読み込んだりできます。結果は常に、バケットごとに同じ最終値になります。

アーカイブ

ソースが新しいデータを収集しなくなったら、ダッシュボードでアーカイブできます。アーカイブすると以降の取り込みが停止します。アーカイブ済みのソースを対象とするリクエストは HTTP 409 "Data source is archived." を返します。

アーカイブは非破壊的です。既存のデータはそのまま残り、そのソースの上に構築されたデータプロファイルは、すでに保存されているデータに対して動作し続けます。
