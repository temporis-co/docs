予測とサンプリング

予測リクエストは、ひとつではなく複数のありうる未来を返します。このページでは、レスポンスの読み方と、サンプリングのコントロールがそれをどう形作るかを説明します。正確なスキーマについてはpredict リファレンスを参照してください。

リクエストの構造

予測の対象となるのは、生のデータソースではなくデータプロファイルです。プロファイルはどの系列を読み取り、どうエンコードするかをすでに知っているので、リクエストはどのプロファイルを使い、どうサンプリングするかだけを伝えればよいのです。4 つのフィールドでそれをカバーします。

data_profilestring必須

あなたのプロファイルのいずれかの名前、たとえば hourly_orders、または公開プロファイル。

countinteger必須

引き出す独立した未来の軌跡の数。最低 1 つ。

temperaturenumber必須

サンプルの散らばり具合。0 より大きくなければならず、約 1.0 が中立。

top_pnumber必須

0 から 1 の範囲の核サンプリングのカットオフ。低くするほど可能性の低い裾を切り落とす。

以下は、hourly_orders プロファイルに対して 20 本の軌跡を要求するリクエストです。

curl https://api.temporis.co/v1/predict \
  -H "Authorization: Bearer ts_your_token" \
  -H "Content-Type: application/json" \
  -d '{
    "data_profile": "hourly_orders",
    "count": 20,
    "temperature": 1.0,
    "top_p": 0.9
  }'
import requests

resp = requests.post(
    "https://api.temporis.co/v1/predict",
    headers={"Authorization": "Bearer ts_your_token"},
    json={
        "data_profile": "hourly_orders",
        "count": 20,
        "temperature": 1.0,
        "top_p": 0.9,
    },
)
resp.raise_for_status()
result = resp.json()
const resp = await fetch("https://api.temporis.co/v1/predict", {
  method: "POST",
  headers: {
    "Authorization": "Bearer ts_your_token",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    data_profile: "hourly_orders",
    count: 20,
    temperature: 1.0,
    top_p: 0.9,
  }),
});
const result = await resp.json();

完全なリファレンス、すなわちすべてのフィールド、ステータスコード、エラーメッセージはpredict エンドポイントリファレンスにあります。

レスポンスの読み方

呼び出しに成功すると、4 つのトップレベルフィールドを伴う HTTP 200 が返されます。

request_idstring

このリクエストの 16 進数の識別子。サポートやご自身のログのために保管してください。

data_profilestring

要求したプロファイルをそのまま返します。

predict_frominteger

予測がそこから続く、最後に観測されたタイムスタンプの Unix 秒。すべての軌跡はこの時点の直後から始まります。

predictionsarray

サンプリングされた軌跡 1 本につき 1 エントリ（長さは count に等しい）。それぞれが loss と、予測行の data 配列を持ちます。

以下の例は、orders の予測から得られた 2 本の軌跡を示しています。プロファイルの interval は 3600、patch_spacing は 2 なので、行は interval × patch_spacing = 7200 秒（2 時間）ごとにステップします。

{
  "request_id": "9f3c1a7e2b4d",
  "data_profile": "hourly_orders",
  "predict_from": 1718888400,
  "predictions": [
    {
      "loss": 0.84,
      "data": [
        { "timestamp": 1718895600, "orders": 142.0 },
        { "timestamp": 1718902800, "orders": 137.5 }
      ]
    },
    {
      "loss": 1.07,
      "data": [
        { "timestamp": 1718895600, "orders": 151.2 },
        { "timestamp": 1718902800, "orders": 129.8 }
      ]
    }
  ]
}

各行は timestamp に加えて、プロファイルが読み取る系列ごとに 1 つのフィールドを持ちます。ここでは orders だけです。タイムスタンプは interval × patch_spacing 秒ごとにステップし、その間隔はすべての軌跡で同じなので、サンプルを行ごとに揃えられます。

サンプリングのコントロール

3 つのパラメータが、返ってくる未来の集合を形作ります。これらは、解像度と多様性を、コストと安定性とのあいだでトレードオフします。

パラメータ働き典型的な値上げ下げするのはどんなとき
count引き出す独立した軌跡の数。サンプルが多いほど分布をよく解像する。20 から 100より滑らかな区間が欲しいときは上げ、トークンコストを抑えたいときは下げる。
temperatureサンプルの散らばり具合。高いほど多様で不確実、低いほど最も可能性の高い経路の近くに集まる。約 1.0予測が狭すぎるときは上げ、非現実的に散らばるときは下げる。
top_p核サンプリングのカットオフ。サンプリング前に可能性の低い裾を切り落とす。0.9 から 1.0意外な飛躍を抑えたいときは下げ、分布全体を保ちたいときは 1.0 に近づける。
コストは count に比例します

各サンプルは独立して生成され、モデル使用のトークンごとに課金されるので、count が 100 のリクエストは count が 20 のおよそ 5 倍のコストがかかります。必要な解像度とコストのバランスをとって count を選んでください。

Loss: 軌跡のランク付け

すべての予測には loss が付随します。値が低いほど、モデルがその軌跡をより典型的、すなわち履歴を踏まえて予期したものにより合致していると考えたことを意味します。loss が高い経路も依然として有効なサンプルであり、ただ中心からより外れているだけです。

loss を使えば、代表となる単一の軌跡を選んだり、集合を要約するときの重みとして使ったりできます。最も典型的な経路を取り出すには次のようにします。

best = min(result["predictions"], key=lambda p: p["loss"])

多変量の予測

プロファイルが複数の系列名を読み取る場合、各予測行は名前ごとに 1 つのフィールドを持ちます。系列は同時に予測されるので、行内で値の整合性が保たれます。kw と price をまとめて読み取るプロファイルは、次のような行を返します。

{ "timestamp": 1718895600, "kw": 482.0, "price": 31.7 }

公開データプロファイル

ご自身でのセットアップなしに、Temporis が維持する共有プロファイルに対して予測できます。公開名を data_profile として渡すだけです。

{ "data_profile": "temporis/polymarket", "count": 20, "temperature": 1.0, "top_p": 0.9 }

レスポンスの形は同一で、各行の系列フィールドは、その公開プロファイルが読み取るものに一致します。
