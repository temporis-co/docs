データの準備

優れた予測は、クリーンで一貫した入力から始まります。このガイドでは、履歴データをモデルが正しく読み取れる形式で Temporis に取り込む方法を説明します。

Unix 秒を一貫して使う

送信するすべてのタイムスタンプは、Unix 秒（1970-01-01 00:00:00 UTC からの経過秒数）を表す整数です。ミリ秒でも ISO 形式の日付文字列でもありません。ソースシステムがミリ秒を返す場合は、送信前に 1000 で割ってください。

タイムゾーンの扱いは 1 つに統一し、決して逸脱しないでください。常に機能する最もシンプルなルールは、すべてを UTC で保存することです。Unix 秒は本質的に UTC なので、送信前に各ローカルタイムスタンプを正しい UTC の瞬間に変換しておけば、系列は内部的に一貫した状態を保てます。同じ系列の中でローカル時刻と UTC の値を混在させると、データ点が静かにずれてしまい、モデルが学習する日次パターンが壊れてしまいます。

タイムスタンプはきりのよい数値に揃える必要はありません。データプロファイルの interval が観測値を規則的なグリッドにスナップします。hourly_orders プロファイルの場合、各データ点は最も近い時刻にバケット化されます。あなたは各観測が実際に発生した瞬間を渡すだけで、整列はプロファイルが処理します。

手早い変換例

Python では、int(datetime(2026, 6, 1, tzinfo=timezone.utc).timestamp()) によって明示的な UTC の瞬間に対する Unix 秒が得られます。変換が曖昧にならないよう、常にタイムゾーンを付与してください。

履歴をバッチで送信する

送信する観測値が 1 件だけということはほとんどなく、数か月から数年分のデータがあるはずです。取り込みエンドポイントはリクエストごとに多数のレコードを受け付けるため、バックフィルを数千レコードずつのバッチに分割し、1 レコードずつではなくループで投稿してください。

取り込みは冪等です。アップサートのキーは (timestamp, name) なので、すでに送信済みのレコードを再送しても、同じ値で上書きされるだけ（何も起こらない）です。つまりバックフィルは安全に再実行できます。バッチが途中で失敗したりスクリプトがクラッシュしたりしても、もう一度実行するだけで構いません。重複が生じることはありません。

import os, requests

token = os.environ["TEMPORIS_TOKEN"]
BASE = "https://api.temporis.co"

# history is a list of (unix_seconds, value) pairs, oldest first
def records_for(history):
    return [{"timestamp": ts, "name": "orders", "value": v} for ts, v in history]

def chunked(seq, size):
    for i in range(0, len(seq), size):
        yield seq[i:i + size]

records = records_for(history)
for batch in chunked(records, 2000):
    resp = requests.post(
        f"{BASE}/v1/data_sources/ingest",
        headers={"Authorization": f"Bearer {token}"},
        json={"data_source": "store_metrics", "records": batch},
    )
    resp.raise_for_status()  # 204 No Content on success
print(f"ingested {len(records)} records")

取り込みが成功すると、本文が空の HTTP 204 No Content が返されるため、解析するものは何もありません。ステータスを確認するだけで十分です。リクエストの完全な形式については取り込みリファレンスを参照してください。

モデルに十分な履歴を与える

プロファイルは準備完了（ready）になるまで予測を提供できません。Temporis は、系列を読み取る際に用いる正規化統計量を計算するために十分な履歴を必要とします。さらに、各予測には少なくとも min_patches 分の直近のコンテキストが必要で、これに満たない場合リクエストは 409「Not enough data for this data profile.」 を返します。

実用的なルールは、迷ったらより多くの履歴を送ることです。履歴が多いほどプロファイルは準備完了になりやすく、min_patches を余裕をもって満たし、モデルが学習できる季節サイクルも増えます。コンテキストウィンドウが使う以上に多く送ってもペナルティはありません。num_patches を超える古いデータ点は、予測時に単に読み込まれないだけです。

min_patches、num_patches、そして「準備完了」が何を意味し、interval とどう関係するかについては、データプロファイルをお読みください。

不規則なタイムスタンプと欠損

現実のデータは雑然としています。センサーが読み取りをスキップしたり、ジョブが数分遅れて実行されたり、店舗が祝日に休業したりします。これらを手作業でクリーニングする必要はありません。プロファイルのリサンプリング処理が対応します。

interval は観測値を規則的なグリッドにバケット化し、わずかに不規則なタイムスタンプを吸収します。10:58 に届いた読み取りと 11:03 に届いた読み取りは、どちらも 11:00 のバケットに入ります。欠損したバケットについては、fill_limit が、Temporis が諦めるまでに前方補完する連続した空バケットの数を設定します。短い欠損は橋渡しされますが、長い欠損は穴として残され、それらの行は作り出されるのではなく削除されます。

系列に繰り返し発生する停止時間（何も起こり得ない時間帯）がある場合は、それを欠損データとして扱わないでください。exclude_window がその時間帯をマスクするので、系列が歪むことはありません。これはプロファイルの設定で、データプロファイルで扱っています。

生の観測値はそのまま送信してください。どこに配置されるか、欠損がどう補完されるか、どの停止時間が無視されるかは、プロファイルが行う判断であり、データに焼き込む変換ではありません。

複数系列の整列

プロファイルは複数の系列を同時に読み取ることができます。これらを共同で予測する多変量プロファイルです。これをうまく機能させるには、系列に共通のタイムラインが必要です。各系列の観測値を同じタイムスタンプの集合で送信し、バケット化後に行が揃うようにしてください。

複数の系列を 1 回の取り込みリクエストで送信できます。各タイムスタンプに各系列のレコードを 1 件ずつ含めるだけです。ここでは 2 つの系列 orders と visitors が同じタイムラインを共有しています。

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718841600, "name": "orders",   "value": 42 },
    { "timestamp": 1718841600, "name": "visitors", "value": 530 },
    { "timestamp": 1718845200, "name": "orders",   "value": 39 },
    { "timestamp": 1718845200, "name": "visitors", "value": 504 },
    { "timestamp": 1718848800, "name": "orders",   "value": 51 },
    { "timestamp": 1718848800, "name": "visitors", "value": 612 }
  ]
}

ある系列が、別の系列が存在するタイムスタンプで欠損している場合、そのバケットは他の欠損と同様に fill_limit に従って補完されます。したがって、わずかな取りこぼしであれば整列は崩れませんが、系列が同じ時計で動いているときに最もきれいな共同予測が得られます。

修正とバックフィル

誤った値を見つけましたか。四半期の数値を修正しますか。特別な削除やパッチ適用は必要ありません。修正したレコードを再送するだけです。アップサートのキーは (timestamp, name) なので、既存の timestamp+name に新しい value を送ると、古い値が上書きされます。

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718841600, "name": "orders", "value": 47 }
  ]
}
再実行に副作用はありません

すべての取り込みは冪等なアップサートなので、修正用のバックフィルがすでに送信済みのデータと重なっても安全です。どのデータ点が変わったかを正確に計算するより簡単であれば、影響を受ける範囲全体を再投稿してください。
