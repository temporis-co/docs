데이터 소스

데이터 소스는 원시 관측값이 저장되는 곳입니다. 이 페이지에서는 Temporis의 데이터 모델과 데이터를 올바르게 전송하는 방법을 다룹니다.

데이터 소스는 대시보드에서 한 번 생성한 다음, 관측값이 발생할 때마다 API를 통해 스트리밍하면 됩니다.

데이터 모델

Temporis는 데이터를 행(row) 단위로 저장합니다. 모든 관측값은 하나의 삼중 조합입니다. 즉 언제 측정되었는지, 무엇이 측정되었는지, 그리고 측정한 숫자입니다.

timestamp
초 단위의 Unix 시간(정수). 관측값이 측정된 시점입니다.
name
이 값이 속하는 시계열 — 예를 들어 orders 또는 revenue.
value
숫자. 부동소수점도 가능합니다.

하나의 데이터 소스는 이름이 지정된 여러 시계열을 나란히 담을 수 있습니다. 이들을 구분하는 것이 바로 name 필드입니다. 여러분의 데이터는 다른 모든 고객의 데이터와 격리되어 있습니다 — 데이터 소스는 오롯이 여러분만의 것입니다.

데이터 소스 생성하기

데이터 소스는 대시보드의 Data Sources에서 생성합니다. 이름을 지정하면 데이터를 받을 준비가 끝납니다.

이름에는 몇 가지 규칙이 있습니다:

  • 소문자로 시작합니다.
  • 소문자, 숫자, 밑줄만 사용합니다.
  • 최대 48자까지 가능합니다.

따라서 store_metrics와 grid_load_eu는 유효하지만, Store-Metrics와 2024data는 그렇지 않습니다. 새로 생성된 데이터 소스는 관측값을 전송하기 전까지 비어 있습니다.

관측값 전송하기

데이터는 ingest 엔드포인트로 추가합니다. 각 호출은 대상 data_source와 records 목록을 함께 전달합니다. 여기서는 몇 개의 orders 측정값을 store_metrics로 전송합니다.

curl -X POST https://api.temporis.co/v1/data_sources/ingest \
  -H "Authorization: Bearer ts_<token>" \
  -H "Content-Type: application/json" \
  -d '{
    "data_source": "store_metrics",
    "records": [
      { "timestamp": 1718870400, "name": "orders", "value": 42 },
      { "timestamp": 1718874000, "name": "orders", "value": 51 },
      { "timestamp": 1718877600, "name": "orders", "value": 47 }
    ]
  }'
import requests

requests.post(
    "https://api.temporis.co/v1/data_sources/ingest",
    headers={"Authorization": "Bearer ts_<token>"},
    json={
        "data_source": "store_metrics",
        "records": [
            {"timestamp": 1718870400, "name": "orders", "value": 42},
            {"timestamp": 1718874000, "name": "orders", "value": 51},
            {"timestamp": 1718877600, "name": "orders", "value": 47},
        ],
    },
)  # -> 204 No Content
await fetch("https://api.temporis.co/v1/data_sources/ingest", {
  method: "POST",
  headers: {
    "Authorization": "Bearer ts_<token>",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    data_source: "store_metrics",
    records: [
      { timestamp: 1718870400, name: "orders", value: 42 },
      { timestamp: 1718874000, name: "orders", value: 51 },
      { timestamp: 1718877600, name: "orders", value: 47 },
    ],
  }),
}); // -> 204 No Content

ingest가 성공하면 빈 본문과 함께 HTTP 204 No Content를 반환합니다 — 읽어 들일 것이 없으며, 행이 그대로 저장될 뿐입니다. 전체 요청 및 오류 세부 정보는 ingest 레퍼런스를 참고하세요.

하나의 소스에 여러 시계열 담기

소스당 하나의 시계열로 제한되지 않습니다. 동일한 data_source에 서로 다른 name 값을 전송하면 별개의 시계열로 공존합니다. 어떤 매장은 주문, 매출, 방문자 수를 모두 store_metrics 안에서 추적할 수 있습니다.

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718870400, "name": "orders",  "value": 42 },
    { "timestamp": 1718870400, "name": "revenue", "value": 1284.50 }
  ]
}

관련된 시계열을 함께 보관하면 편리할 뿐 아니라, 나중에 이들을 공동으로 예측할 수 있습니다. 데이터 프로필은 이러한 이름 중 실제로 어떤 것을 읽을지 선택하는 곳입니다.

업데이트와 백필

업서트(upsert) 키는 (timestamp, name) 쌍입니다. 이미 존재하는 timestamp와 name을 가진 레코드를 전송하면, 중복을 추가하는 대신 저장된 값을 덮어씁니다.

백필과 재시도는 안전합니다

쓰기 작업이 (timestamp, name)에 대해 멱등(idempotent)이므로, 네트워크 장애 후 배치를 다시 보내거나, 잘못된 측정값을 정정하거나, 수개월치 이력을 어떤 순서로든 로드해도 됩니다 — 결과는 항상 버킷별로 동일한 최종 값입니다.

보관 처리

소스가 더 이상 새 데이터를 수집하지 않을 때는 대시보드에서 보관 처리(archive)할 수 있습니다. 보관 처리하면 추가 수집이 중단됩니다. 보관된 소스를 대상으로 한 요청은 HTTP 409 "Data source is archived."를 반환합니다.

보관 처리는 비파괴적입니다. 기존 데이터는 그대로 유지되며, 해당 소스를 기반으로 구축된 데이터 프로필은 이미 저장된 데이터에 대해 계속 작동합니다.
