Previsões e amostragem

Uma requisição de previsão retorna vários futuros possíveis em vez de um só. Esta página explica como ler a resposta e como os controles de amostragem a moldam. Para o esquema exato, consulte a referência de predição.

Anatomia de uma requisição

Você faz previsões contra um perfil de dados, não contra uma fonte de dados bruta. O perfil já sabe quais séries ler e como codificá-las, então uma requisição só precisa dizer qual perfil e como amostrar. Quatro campos cobrem isso:

data_profilestringObrigatório

O nome de um dos seus perfis, como hourly_orders — ou um perfil público.

countintegerObrigatório

Quantas trajetórias futuras independentes sortear. No mínimo 1.

temperaturenumberObrigatório

Quão espalhadas as amostras ficam. Deve ser maior que 0; cerca de 1.0 é neutro.

top_pnumberObrigatório

Corte de amostragem por núcleo na faixa de 0 a 1. Valores mais baixos cortam a cauda improvável.

Aqui está uma requisição de 20 trajetórias contra o perfil hourly_orders:

curl https://api.temporis.co/v1/predict \
  -H "Authorization: Bearer ts_your_token" \
  -H "Content-Type: application/json" \
  -d '{
    "data_profile": "hourly_orders",
    "count": 20,
    "temperature": 1.0,
    "top_p": 0.9
  }'
import requests

resp = requests.post(
    "https://api.temporis.co/v1/predict",
    headers={"Authorization": "Bearer ts_your_token"},
    json={
        "data_profile": "hourly_orders",
        "count": 20,
        "temperature": 1.0,
        "top_p": 0.9,
    },
)
resp.raise_for_status()
result = resp.json()
const resp = await fetch("https://api.temporis.co/v1/predict", {
  method: "POST",
  headers: {
    "Authorization": "Bearer ts_your_token",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    data_profile: "hourly_orders",
    count: 20,
    temperature: 1.0,
    top_p: 0.9,
  }),
});
const result = await resp.json();

A referência completa — cada campo, código de status e mensagem de erro — está em a referência do endpoint de predição.

Lendo a resposta

Uma chamada bem-sucedida retorna HTTP 200 com quatro campos de nível superior:

request_idstring

Um identificador hexadecimal para esta requisição. Guarde-o para suporte e seus próprios registros.

data_profilestring

Ecoa o perfil que você solicitou.

predict_frominteger

Segundos Unix do último timestamp observado a partir do qual a previsão continua. Toda trajetória começa logo após esse ponto.

predictionsarray

Uma entrada por trajetória amostrada (o comprimento é igual a count). Cada uma carrega um loss e um array data de linhas de previsão.

O exemplo abaixo mostra duas das trajetórias de uma previsão de orders. O interval do perfil é 3600 e seu patch_spacing é 2, então as linhas avançam de interval × patch_spacing = 7200 segundos (duas horas):

{
  "request_id": "9f3c1a7e2b4d",
  "data_profile": "hourly_orders",
  "predict_from": 1718888400,
  "predictions": [
    {
      "loss": 0.84,
      "data": [
        { "timestamp": 1718895600, "orders": 142.0 },
        { "timestamp": 1718902800, "orders": 137.5 }
      ]
    },
    {
      "loss": 1.07,
      "data": [
        { "timestamp": 1718895600, "orders": 151.2 },
        { "timestamp": 1718902800, "orders": 129.8 }
      ]
    }
  ]
}

Cada linha tem um timestamp mais um campo por série que o perfil lê — aqui, apenas orders. Os timestamps avançam de interval × patch_spacing segundos, com o mesmo espaçamento para cada trajetória, de modo que você pode alinhar as amostras linha a linha.

Os controles de amostragem

Três parâmetros moldam o conjunto de futuros que você recebe de volta. Eles equilibram resolução e diversidade contra custo e estabilidade.

ParâmetroO que fazValor típicoAumente / diminua quando
countNúmero de trajetórias independentes sorteadas. Mais amostras resolvem melhor a distribuição.20 a 100Aumente para intervalos mais suaves; diminua para cortar o custo em tokens.
temperatureQuão espalhadas as amostras ficam. Mais alto é mais diverso e incerto; mais baixo agrupa perto do caminho mais provável.~1.0Aumente quando as previsões parecerem estreitas demais; diminua quando se dispersarem de forma irreal.
top_pCorte por núcleo. Corta a cauda improvável antes da amostragem.0.9 a 1.0Diminua para suprimir saltos surpreendentes; aumente em direção a 1.0 para manter a distribuição completa.
O custo escala com count

Cada amostra é gerada de forma independente e tarifada por token de uso do modelo, então uma requisição com count 100 custa aproximadamente cinco vezes mais do que uma com count 20. Escolha count para equilibrar a resolução de que você precisa com o custo.

Loss: classificando trajetórias

Toda previsão carrega um loss. Mais baixo significa que o modelo considerou aquela trajetória mais típica — mais de acordo com o que ele esperava dado o histórico. Caminhos de loss mais alto ainda são amostras válidas; eles são apenas menos centrais.

Use loss para escolher uma única trajetória representativa, ou como peso quando você resumir o conjunto. Para pegar o caminho mais típico:

best = min(result["predictions"], key=lambda p: p["loss"])

Previsões multivariadas

Se um perfil lê vários nomes de séries, cada linha de previsão carrega um campo por nome — as séries são previstas em conjunto, de modo que seus valores permanecem consistentes dentro de uma linha. Um perfil que lê kw e price juntos retorna linhas como:

{ "timestamp": 1718895600, "kw": 482.0, "price": 31.7 }

Perfis de dados públicos

Você pode fazer previsões contra perfis compartilhados que o Temporis mantém, sem nenhuma configuração própria — basta passar o nome público como data_profile:

{ "data_profile": "temporis/polymarket", "count": 20, "temperature": 1.0, "top_p": 0.9 }

O formato da resposta é idêntico; os campos de série em cada linha correspondem ao que quer que o perfil público leia.
