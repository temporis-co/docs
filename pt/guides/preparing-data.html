Preparando seus dados

Boas previsões começam com uma entrada limpa e consistente. Este guia mostra como colocar seu histórico no Temporis em um formato que ele consegue ler bem.

Use segundos Unix, de forma consistente

Cada timestamp que você envia é uma contagem inteira de segundos Unix — o número de segundos desde 1970-01-01 00:00:00 UTC. Não milissegundos, nem uma string de data ISO. Se o seu sistema de origem fornece milissegundos, divida por 1000 antes de enviar.

Escolha uma disciplina de fuso horário e nunca se desvie dela. A regra mais simples que sempre funciona: armazene tudo em UTC. Segundos Unix são inerentemente UTC, então, desde que você converta cada timestamp local para o seu verdadeiro instante UTC antes de enviar, sua série permanece internamente consistente. Misturar valores em horário local e UTC na mesma série desloca pontos silenciosamente e corrompe o padrão diário que o modelo aprende.

Seus timestamps não precisam cair em números redondos. O interval de um perfil de dados encaixa as observações em uma grade regular — para o perfil hourly_orders, cada ponto é agrupado na hora mais próxima. Você fornece o instante real em que cada observação aconteceu; o perfil cuida do alinhamento.

Uma conversão rápida

Em Python, int(datetime(2026, 6, 1, tzinfo=timezone.utc).timestamp()) fornece os segundos Unix para um instante UTC explícito. Sempre anexe um fuso horário para que a conversão seja inequívoca.

Envie o histórico em lotes

Raramente você tem apenas uma observação para enviar — você tem meses ou anos delas. O endpoint de ingestão aceita muitos registros por requisição, então divida seu backfill em lotes de alguns milhares de registros e os envie em um laço, em vez de um registro por vez.

A ingestão é idempotente. A chave de upsert é (timestamp, name), então reenviar um registro que você já enviou simplesmente o sobrescreve com o mesmo valor — uma operação sem efeito. Isso significa que um backfill é seguro para reexecutar: se um lote falhar pela metade ou seu script travar, basta iniciá-lo novamente. Você não criará duplicatas.

import os, requests

token = os.environ["TEMPORIS_TOKEN"]
BASE = "https://api.temporis.co"

# history is a list of (unix_seconds, value) pairs, oldest first
def records_for(history):
    return [{"timestamp": ts, "name": "orders", "value": v} for ts, v in history]

def chunked(seq, size):
    for i in range(0, len(seq), size):
        yield seq[i:i + size]

records = records_for(history)
for batch in chunked(records, 2000):
    resp = requests.post(
        f"{BASE}/v1/data_sources/ingest",
        headers={"Authorization": f"Bearer {token}"},
        json={"data_source": "store_metrics", "records": batch},
    )
    resp.raise_for_status()  # 204 No Content on success
print(f"ingested {len(records)} records")

Uma ingestão bem-sucedida retorna HTTP 204 No Content com corpo vazio, então não há nada para analisar — apenas confirme o status. Veja a referência de ingestão para o formato completo da requisição.

Dê histórico suficiente ao modelo

Um perfil não pode servir previsões até estar pronto — o Temporis precisa de histórico suficiente para calcular as estatísticas de normalização através das quais ele lê sua série. E cada previsão precisa de pelo menos min_patches de contexto recente, ou a requisição retorna 409 "Not enough data for this data profile."

A regra prática: na dúvida, envie mais histórico. Mais histórico permite que um perfil fique pronto, ultrapassa confortavelmente o min_patches e dá ao modelo mais ciclos sazonais para aprender. Não há penalidade por enviar mais do que a janela de contexto utiliza — pontos mais antigos além de num_patches simplesmente não são lidos no momento da previsão.

Para entender o que min_patches, num_patches e "pronto" significam e como eles se relacionam com o seu interval, leia Perfis de dados.

Timestamps irregulares e lacunas

Dados reais são bagunçados. Sensores pulam uma leitura, um job roda alguns minutos atrasado, uma loja fica fechada em um feriado. Você não precisa limpar nada disso manualmente — a etapa de reamostragem do perfil cuida disso.

O interval agrupa suas observações em uma grade regular, o que absorve timestamps ligeiramente irregulares: uma leitura que chega às 10:58 e outra às 11:03 caem ambas no balde das 11:00. Para baldes ausentes, o fill_limit define quantos baldes vazios consecutivos o Temporis preencherá para frente antes de desistir. Uma lacuna curta é preenchida; uma lacuna longa é deixada como um buraco e essas linhas são descartadas em vez de inventadas.

Se sua série tem tempo morto recorrente — horas em que nada pode possivelmente acontecer — não trate isso como dados ausentes de forma alguma. Um exclude_window mascara essa faixa de horário do dia para que ela nunca distorça a série. Essa é uma configuração de perfil, abordada em Perfis de dados.

Você envia as observações brutas como as tem. Onde elas caem, como as lacunas são preenchidas e qual tempo morto é ignorado são decisões que o perfil toma — não transformações que você embute nos dados.

Alinhando várias séries

Um perfil pode ler várias séries de uma vez — um perfil multivariado que as prevê em conjunto. Para que isso funcione bem, as séries precisam de uma linha do tempo comum: envie a observação de cada série no mesmo conjunto de timestamps para que as linhas se alinhem após o agrupamento.

Você pode enviar várias séries em uma única requisição de ingestão — basta incluir um registro por série em cada timestamp. Aqui, duas séries, orders e visitors, compartilham a mesma linha do tempo:

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718841600, "name": "orders",   "value": 42 },
    { "timestamp": 1718841600, "name": "visitors", "value": 530 },
    { "timestamp": 1718845200, "name": "orders",   "value": 39 },
    { "timestamp": 1718845200, "name": "visitors", "value": 504 },
    { "timestamp": 1718848800, "name": "orders",   "value": 51 },
    { "timestamp": 1718848800, "name": "visitors", "value": 612 }
  ]
}

Se uma série está ausente em um timestamp onde outra está presente, esse balde é preenchido conforme o fill_limit, como qualquer outra lacuna — então alguns retardatários não quebrarão o alinhamento, mas você obterá a previsão conjunta mais limpa quando as séries andarem no mesmo relógio.

Correções e backfills

Encontrou um valor errado? Reformulando um trimestre? Você não exclui nem corrige nada de especial — basta reenviar o registro corrigido. Como a chave de upsert é (timestamp, name), enviar um novo value em um timestamp+name existente sobrescreve o antigo.

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718841600, "name": "orders", "value": 47 }
  ]
}
Reexecutar não tem efeitos colaterais

Como toda ingestão é um upsert idempotente, um backfill corrigido pode sobrepor com segurança dados que você já enviou. Reenvie toda a faixa afetada se for mais fácil do que calcular exatamente quais pontos mudaram.
