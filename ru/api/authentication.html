Аутентификация

Каждый запрос к API Temporis аутентифицируется с помощью bearer-токена. Здесь описано, как его создать, отправить и обезопасить.

Токены доступа

Токены доступа создаются в панели управления, в разделе Access Tokens. Токен — это длинная случайная строка с префиксом ts_.

Полный токен показывается один раз — в момент его создания. Temporis хранит на сервере только его хеш, но не сам токен. Это означает, что утраченный токен невозможно восстановить: если вы его потеряли, удалите его и создайте новый.

Относитесь к токенам как к паролям

Любой, у кого есть один из ваших токенов, может загружать данные и запускать предсказания — а предсказания тарифицируются по вашему аккаунту. Берегите их соответствующим образом.

Аутентификация запроса

Отправляйте токен в заголовке Authorization в каждом запросе, используя схему Bearer:

Authorization: Bearer ts_your_token_here

Во всех примерах этой документации токен читается из переменной окружения TEMPORIS_TOKEN, поэтому секрет никогда не появляется в исходном коде. Один раз экспортируйте его в своей оболочке:

export TEMPORIS_TOKEN="ts_your_token_here"

Затем добавьте его в виде заголовка:

curl https://api.temporis.co/v1/predict \
  -H "Authorization: Bearer $TEMPORIS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{ "data_profile": "hourly_orders", "count": 5, "temperature": 1.0, "top_p": 0.9 }'
import os, requests

token = os.environ["TEMPORIS_TOKEN"]

resp = requests.post(
    "https://api.temporis.co/v1/predict",
    headers={"Authorization": f"Bearer {token}"},
    json={"data_profile": "hourly_orders", "count": 5, "temperature": 1.0, "top_p": 0.9},
)
resp.raise_for_status()
const token = process.env.TEMPORIS_TOKEN;

const resp = await fetch("https://api.temporis.co/v1/predict", {
  method: "POST",
  headers: {
    "Authorization": `Bearer ${token}`,
    "Content-Type": "application/json",
  },
  body: JSON.stringify({ data_profile: "hourly_orders", count: 5, temperature: 1.0, top_p: 0.9 }),
});

Когда аутентификация не удаётся

Если заголовок Authorization отсутствует или имеет неверный формат либо токен неизвестен, API отвечает HTTP 401 и JSON-телом, описывающим, что пошло не так:

{ "detail": "Invalid access token." }

Отсутствующий или некорректный заголовок вместо этого возвращает { "detail": "Missing access token." }. Оба случая дают 401. Полный список ответов см. в разделе Ошибки и коды состояния.

Как обезопасить токены

Несколько привычек помогут уберечь ваши токены от попадания в чужие руки:

  • Никогда не коммитьте токены в систему контроля версий и никогда не размещайте их в клиентском коде. Браузеры и мобильные приложения поставляют свой исходный код пользователям.
  • Читайте токены из переменных окружения или менеджера секретов. Держите их подальше от логов, скриншотов и тикетов.
  • Создавайте отдельный токен для каждого сервиса, чтобы можно было отозвать один, не нарушая работу остальных.
  • Удалите токен в панели управления, чтобы отозвать его мгновенно. Удаление вступает в силу немедленно — следующий запрос с этим токеном получит 401.
