Источники данных

Источник данных — это место, где хранятся ваши исходные наблюдения. Эта страница охватывает модель данных Temporis и то, как правильно отправлять данные в неё.

Вы создаёте источник данных один раз в панели управления, а затем передаёте в него наблюдения потоком через API по мере их появления.

Модель данных

Temporis хранит данные в виде строк. Каждое наблюдение — это одна тройка: когда что-то было измерено, что было измерено и число, которое вы измерили.

timestamp
Время Unix в секундах (целое число). Момент, когда было сделано наблюдение.
name
Ряд, которому принадлежит это значение — например, orders или revenue.
value
Число. Числа с плавающей точкой допустимы.

Один источник данных может содержать множество именованных рядов бок о бок. Поле name — это то, что их различает. Ваши данные изолированы от данных любого другого клиента — источник данных принадлежит только вам.

Создание источника данных

Вы создаёте источник данных в панели управления в разделе Data Sources. Дайте ему имя — и он готов принимать данные.

Имена подчиняются нескольким правилам:

  • Начинаются со строчной буквы.
  • Используют только строчные буквы, цифры и подчёркивания.
  • До 48 символов.

Так, store_metrics и grid_load_eu допустимы; Store-Metrics и 2024data — нет. Только что созданный источник данных пуст, пока вы не отправите в него наблюдения.

Отправка наблюдений

Вы добавляете данные через эндпоинт приёма данных. Каждый вызов несёт целевой data_source и список records. Здесь мы отправляем несколько показаний orders в store_metrics.

curl -X POST https://api.temporis.co/v1/data_sources/ingest \
  -H "Authorization: Bearer ts_<token>" \
  -H "Content-Type: application/json" \
  -d '{
    "data_source": "store_metrics",
    "records": [
      { "timestamp": 1718870400, "name": "orders", "value": 42 },
      { "timestamp": 1718874000, "name": "orders", "value": 51 },
      { "timestamp": 1718877600, "name": "orders", "value": 47 }
    ]
  }'
import requests

requests.post(
    "https://api.temporis.co/v1/data_sources/ingest",
    headers={"Authorization": "Bearer ts_<token>"},
    json={
        "data_source": "store_metrics",
        "records": [
            {"timestamp": 1718870400, "name": "orders", "value": 42},
            {"timestamp": 1718874000, "name": "orders", "value": 51},
            {"timestamp": 1718877600, "name": "orders", "value": 47},
        ],
    },
)  # -> 204 No Content
await fetch("https://api.temporis.co/v1/data_sources/ingest", {
  method: "POST",
  headers: {
    "Authorization": "Bearer ts_<token>",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    data_source: "store_metrics",
    records: [
      { timestamp: 1718870400, name: "orders", value: 42 },
      { timestamp: 1718874000, name: "orders", value: 51 },
      { timestamp: 1718877600, name: "orders", value: 47 },
    ],
  }),
}); // -> 204 No Content

Успешный приём данных возвращает HTTP 204 No Content с пустым телом — читать обратно нечего, строки просто сохраняются. См. справочник по приёму данных для полного описания запроса и сведений об ошибках.

Множество рядов в одном источнике

Вы не ограничены одним рядом на источник. Отправляйте разные значения name в один и тот же data_source, и они сосуществуют как отдельные ряды. Магазин может отслеживать заказы, выручку и посетителей — всё в store_metrics.

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718870400, "name": "orders",  "value": 42 },
    { "timestamp": 1718870400, "name": "revenue", "value": 1284.50 }
  ]
}

Держать связанные ряды вместе удобно, и это позволяет позже прогнозировать их совместно. Профиль данных — это то место, где вы выбираете, какие из этих имён действительно читать.

Обновления и дозагрузки

Ключ upsert — это пара (timestamp, name). Отправка записи, у которой timestamp и name уже существуют, перезаписывает сохранённое значение, а не добавляет дубликат.

Дозагрузки и повторы безопасны

Поскольку записи идемпотентны по (timestamp, name), вы можете повторно отправить пакет после сбоя сети, исправить ошибочное показание или загрузить месяцы истории в любом порядке — результат всегда даёт одно и то же итоговое значение на корзину.

Архивирование

Когда источник больше не собирает новые данные, вы можете заархивировать его в панели управления. Архивирование прекращает дальнейший приём данных: запрос, нацеленный на заархивированный источник, возвращает HTTP 409 "Data source is archived."

Архивирование не разрушительно. Существующие данные остаются на месте, и любые профили данных, построенные на источнике, продолжают работать с уже сохранёнными данными.
