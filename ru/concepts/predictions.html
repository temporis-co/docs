Прогнозы и сэмплирование

Запрос на прогноз возвращает несколько возможных вариантов будущего, а не один. На этой странице объясняется, как читать ответ и как его формируют средства управления сэмплированием. Точную схему см. в справочнике по прогнозированию.

Анатомия запроса

Вы строите прогноз по профилю данных, а не по необработанному источнику данных. Профиль уже знает, какие ряды читать и как их кодировать, поэтому в запросе нужно лишь указать, какой профиль и как сэмплировать. Это охватывают четыре поля:

data_profilestringОбязательный

Имя одного из ваших профилей, например hourly_orders, или публичного профиля.

countintegerОбязательный

Сколько независимых будущих траекторий построить. Не менее 1.

temperaturenumberОбязательный

Насколько разбросаны сэмплы. Должно быть больше 0; около 1.0 — нейтрально.

top_pnumberОбязательный

Порог nucleus-сэмплирования в диапазоне от 0 до 1. Меньшее значение обрезает маловероятный хвост.

Вот запрос на 20 траекторий по профилю hourly_orders:

curl https://api.temporis.co/v1/predict \
  -H "Authorization: Bearer ts_your_token" \
  -H "Content-Type: application/json" \
  -d '{
    "data_profile": "hourly_orders",
    "count": 20,
    "temperature": 1.0,
    "top_p": 0.9
  }'
import requests

resp = requests.post(
    "https://api.temporis.co/v1/predict",
    headers={"Authorization": "Bearer ts_your_token"},
    json={
        "data_profile": "hourly_orders",
        "count": 20,
        "temperature": 1.0,
        "top_p": 0.9,
    },
)
resp.raise_for_status()
result = resp.json()
const resp = await fetch("https://api.temporis.co/v1/predict", {
  method: "POST",
  headers: {
    "Authorization": "Bearer ts_your_token",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    data_profile: "hourly_orders",
    count: 20,
    temperature: 1.0,
    top_p: 0.9,
  }),
});
const result = await resp.json();

Полный справочник — каждое поле, код состояния и сообщение об ошибке — находится в справочнике по эндпоинту прогнозирования.

Чтение ответа

Успешный вызов возвращает HTTP 200 с четырьмя полями верхнего уровня:

request_idstring

Шестнадцатеричный идентификатор этого запроса. Сохраните его для поддержки и собственных логов.

data_profilestring

Возвращает профиль, который вы запросили.

predict_frominteger

Время в Unix-секундах последней наблюдённой метки времени, от которой продолжается прогноз. Каждая траектория начинается сразу после этой точки.

predictionsarray

По одной записи на каждую построенную траекторию (длина равна count). Каждая содержит loss и массив data строк прогноза.

В примере ниже показаны две из траекторий прогноза orders. У профиля interval равен 3600, а patch_spacing равен 2, поэтому строки идут с шагом interval × patch_spacing = 7200 секунд (два часа):

{
  "request_id": "9f3c1a7e2b4d",
  "data_profile": "hourly_orders",
  "predict_from": 1718888400,
  "predictions": [
    {
      "loss": 0.84,
      "data": [
        { "timestamp": 1718895600, "orders": 142.0 },
        { "timestamp": 1718902800, "orders": 137.5 }
      ]
    },
    {
      "loss": 1.07,
      "data": [
        { "timestamp": 1718895600, "orders": 151.2 },
        { "timestamp": 1718902800, "orders": 129.8 }
      ]
    }
  ]
}

Каждая строка содержит timestamp плюс по одному полю на каждый ряд, который читает профиль, — здесь только orders. Метки времени идут с шагом interval × patch_spacing секунд, с одинаковым шагом для каждой траектории, так что вы можете выравнивать сэмплы строка за строкой.

Средства управления сэмплированием

Три параметра формируют набор вариантов будущего, который вы получаете. Они уравновешивают разрешение и разнообразие с одной стороны и стоимость и стабильность с другой.

ПараметрЧто делаетТипичное значениеКогда повышать / понижать
countЧисло построенных независимых траекторий. Больше сэмплов лучше разрешают распределение.от 20 до 100Повышайте для более гладких интервалов; понижайте, чтобы сократить стоимость в токенах.
temperatureНасколько разбросаны сэмплы. Выше — разнообразнее и неопределённее; ниже — группируются вблизи наиболее вероятного пути.~1.0Повышайте, когда прогнозы выглядят слишком узкими; понижайте, когда они нереалистично разбросаны.
top_pПорог nucleus. Обрезает маловероятный хвост перед сэмплированием.от 0.9 до 1.0Понижайте, чтобы подавить неожиданные скачки; повышайте к 1.0, чтобы сохранить всё распределение.
Стоимость растёт вместе с count

Каждый сэмпл генерируется независимо и тарифицируется по токенам использования модели, поэтому запрос с count 100 стоит примерно в пять раз дороже запроса с count 20. Выбирайте count, чтобы сбалансировать нужное разрешение и стоимость.

Loss: ранжирование траекторий

Каждый прогноз содержит loss. Меньшее значение означает, что модель сочла эту траекторию более типичной — больше соответствующей тому, что она ожидала с учётом истории. Пути с более высоким loss по-прежнему являются допустимыми сэмплами; они просто менее центральны.

Используйте loss, чтобы выбрать одну репрезентативную траекторию, или как вес при обобщении набора. Чтобы взять самый типичный путь:

best = min(result["predictions"], key=lambda p: p["loss"])

Многомерные прогнозы

Если профиль читает несколько имён рядов, каждая строка прогноза содержит по одному полю на каждое имя — ряды прогнозируются совместно, так что их значения остаются согласованными внутри строки. Профиль, читающий kw и price вместе, возвращает строки вида:

{ "timestamp": 1718895600, "kw": 482.0, "price": 31.7 }

Публичные профили данных

Вы можете строить прогноз по общим профилям, которые поддерживает Temporis, без какой-либо собственной настройки — просто передайте публичное имя в data_profile:

{ "data_profile": "temporis/polymarket", "count": 20, "temperature": 1.0, "top_p": 0.9 }

Форма ответа идентична; поля рядов в каждой строке соответствуют тому, что читает публичный профиль.
