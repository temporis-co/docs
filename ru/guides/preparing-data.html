Подготовка данных

Хорошие прогнозы начинаются с чистых и согласованных входных данных. В этом руководстве описано, как загрузить историю в Temporis в виде, который модель хорошо читает.

Используйте Unix-секунды единообразно

Каждая отправляемая вами временная метка — это целочисленное количество Unix-секунд, то есть число секунд, прошедших с 1970-01-01 00:00:00 UTC. Не миллисекунды и не строка с датой в формате ISO. Если ваша исходная система выдаёт миллисекунды, разделите их на 1000 перед отправкой.

Выберите единую дисциплину работы с часовыми поясами и никогда от неё не отклоняйтесь. Простейшее правило, которое всегда работает: храните всё в UTC. Unix-секунды по своей сути выражены в UTC, поэтому, пока вы преобразуете каждую локальную метку в её истинный момент в UTC перед отправкой, ваш ряд остаётся внутренне согласованным. Смешивание значений в локальном времени и в UTC в одном ряду незаметно сдвигает точки и портит суточный паттерн, который изучает модель.

Ваши временные метки не обязаны приходиться на круглые числа. Параметр interval профиля данных привязывает наблюдения к регулярной сетке — для профиля hourly_orders каждая точка распределяется в ближайший час. Вы предоставляете реальный момент, когда произошло каждое наблюдение, а выравниванием занимается профиль.

Быстрое преобразование

В Python выражение int(datetime(2026, 6, 1, tzinfo=timezone.utc).timestamp()) даёт Unix-секунды для явно заданного момента в UTC. Всегда указывайте часовой пояс, чтобы преобразование было однозначным.

Отправляйте историю пакетами

Редко бывает, что нужно отправить лишь одно наблюдение — у вас их за месяцы или годы. Эндпоинт загрузки принимает много записей за один запрос, поэтому разбивайте заполнение истории на пакеты по несколько тысяч записей и отправляйте их в цикле, а не по одной записи за раз.

Загрузка идемпотентна. Ключ для upsert — это (timestamp, name), поэтому повторная отправка уже отправленной записи просто перезаписывает её тем же значением — это пустая операция. Это значит, что заполнение истории безопасно повторять: если пакет не прошёл на середине или ваш скрипт упал, просто запустите его снова. Дубликатов не возникнет.

import os, requests

token = os.environ["TEMPORIS_TOKEN"]
BASE = "https://api.temporis.co"

# history is a list of (unix_seconds, value) pairs, oldest first
def records_for(history):
    return [{"timestamp": ts, "name": "orders", "value": v} for ts, v in history]

def chunked(seq, size):
    for i in range(0, len(seq), size):
        yield seq[i:i + size]

records = records_for(history)
for batch in chunked(records, 2000):
    resp = requests.post(
        f"{BASE}/v1/data_sources/ingest",
        headers={"Authorization": f"Bearer {token}"},
        json={"data_source": "store_metrics", "records": batch},
    )
    resp.raise_for_status()  # 204 No Content on success
print(f"ingested {len(records)} records")

Успешная загрузка возвращает HTTP 204 No Content с пустым телом, так что разбирать нечего — достаточно убедиться в статусе. Полное описание формата запроса см. в справочнике по загрузке данных.

Дайте модели достаточно истории

Профиль не может выдавать прогнозы, пока он не готов: Temporis нужно достаточно истории, чтобы вычислить статистику нормализации, через которую он читает ваш ряд. А каждому прогнозу требуется как минимум min_patches недавнего контекста, иначе запрос вернёт 409 «Not enough data for this data profile.»

Практическое правило: если сомневаетесь — отправляйте больше истории. Больше истории позволяет профилю стать готовым, с запасом перекрывает min_patches и даёт модели больше сезонных циклов для обучения. За отправку большего, чем использует контекстное окно, штрафа нет — точки старше, чем покрывает num_patches, просто не читаются при прогнозировании.

Чтобы понять, что означают min_patches, num_patches и «готовность» и как они связаны с вашим interval, прочитайте Профили данных.

Нерегулярные метки и пропуски

Реальные данные неаккуратны. Датчик пропускает измерение, задача запускается с опозданием на несколько минут, магазин закрыт в праздник. Чистить всё это вручную не нужно — за это отвечает шаг ресэмплинга в профиле.

Параметр interval распределяет ваши наблюдения по регулярной сетке, что поглощает слегка нерегулярные метки: измерение, пришедшее в 10:58, и измерение в 11:03 оба попадают в корзину 11:00. Для отсутствующих корзин fill_limit задаёт, сколько подряд идущих пустых корзин Temporis заполнит протягиванием вперёд, прежде чем сдаться. Короткий пропуск перекрывается; длинный пропуск остаётся дырой, и эти строки отбрасываются, а не выдумываются.

Если в вашем ряду есть повторяющиеся периоды простоя — часы, когда заведомо ничего не может произойти, — не считайте их пропущенными данными вовсе. Параметр exclude_window маскирует этот диапазон времени суток, чтобы он никогда не искажал ряд. Это настройка профиля, описанная в Профилях данных.

Вы отправляете сырые наблюдения в том виде, в каком они у вас есть. Куда они попадут, как заполняются пропуски и какое время простоя игнорируется — это решения, которые принимает профиль, а не преобразования, которые вы запекаете в данные.

Выравнивание нескольких рядов

Профиль может читать несколько рядов сразу — многомерный профиль, прогнозирующий их совместно. Чтобы это хорошо работало, рядам нужна общая временная шкала: отправляйте наблюдения каждого ряда на одном и том же наборе временных меток, чтобы строки совпали после распределения по корзинам.

Несколько рядов можно отправить в одном запросе на загрузку — просто включите по записи для каждого ряда на каждую временную метку. Здесь два ряда, orders и visitors, разделяют одну временную шкалу:

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718841600, "name": "orders",   "value": 42 },
    { "timestamp": 1718841600, "name": "visitors", "value": 530 },
    { "timestamp": 1718845200, "name": "orders",   "value": 39 },
    { "timestamp": 1718845200, "name": "visitors", "value": 504 },
    { "timestamp": 1718848800, "name": "orders",   "value": 51 },
    { "timestamp": 1718848800, "name": "visitors", "value": 612 }
  ]
}

Если один ряд отсутствует на временной метке, где другой присутствует, эта корзина заполняется по fill_limit, как любой другой пропуск, — поэтому несколько отставших значений не нарушат выравнивание, но самый чистый совместный прогноз вы получите, когда ряды движутся по одним и тем же часам.

Исправления и дозаполнения

Нашли неверное значение? Пересчитываете квартал? Вам не нужно ничего удалять или особым образом исправлять — просто отправьте исправленную запись повторно. Поскольку ключ для upsert — это (timestamp, name), отправка нового value на существующей паре timestamp+name перезаписывает старое.

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718841600, "name": "orders", "value": 47 }
  ]
}
Повторный запуск не имеет побочных эффектов

Поскольку каждая загрузка — это идемпотентный upsert, исправленное дозаполнение может безопасно перекрываться с уже отправленными данными. Переотправьте весь затронутый диапазон, если это проще, чем вычислять, какие именно точки изменились.
