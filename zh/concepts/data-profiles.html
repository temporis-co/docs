文档/概念/数据画像

数据画像

数据画像是告诉 Temporis 如何读取数据源的配方：读取哪些序列、以何种节奏、使用多少历史，以及如何清洗它。它就是你用来预测的对象。

数据画像定义了什么

数据源存放的是原始的行。数据画像会把这些行变成一个干净、间距规则、模型可以读取的信号——挑选序列、把观测值对齐到时间网格、决定使用多少历史，并整理空缺。当你调用 predict 时，你指定的是一个数据画像，而不是一个数据源。

你在控制台中创建和编辑数据画像。这里的示例是 hourly_orders：它以一小时为网格（interval 为 3600）读取 store_metrics 中的 orders 序列。

选择数据

数据源与序列

有两个设置决定了数据画像读取什么：

  • data_source —— 从哪个数据源读取。
  • names —— 该数据源中的一个或多个序列名称。

只列出单个 name 会得到单变量预测。列出多个——比如 orders 和 visitors——会使该数据画像成为多变量的：这些序列被联合预测，因此模型可以利用它们之间的关系。

时间网格

真实的观测值很少以完全均匀的节奏到达，因此数据画像会将它们重采样到一个规则的网格上。

interval 是每个网格桶的大小，以秒为单位。当 interval 为 3600 时，观测值会被对齐到按小时划分的桶中。网格以你最近的观测值为锚点，并从那里向前回溯，因此最新的数据总能干净地落在网格上。

模型并不会逐点读取每一个点。它以 patch 为单位工作——也就是序列中固定大小的块：

  • patch_len —— 多少个点构成一个 patch。
  • patch_spacing —— 它所用各点之间的步长，是一种下采样方式。
  • num_patches —— 作为上下文最多使用多少个历史 patch。
  • min_patches —— 提供一次预测所需的最少历史 patch 数。

直观地说：模型看到的上下文大致是 num_patches 个近期历史 patch。如果某个数据画像可用的 patch 少于 min_patches，它就无法提供服务，你会得到 HTTP 409 "Not enough data for this data profile."

步长还决定了你的预测分辨率。预测点之间的有效步长为 interval × patch_spacing 秒。当 interval 为 3600、patch_spacing 为 1 时，预测点之间相隔一小时。

清洗序列

重采样之后，某些桶可能没有观测值。有两个设置决定如何处理这种情况：

  • fill_limit —— 会被前向填充（把最后一个已知值向前延续）的最长连续空桶段。比这更长的空缺会被视为真正缺失，那些行会被丢弃。
  • exclude_window —— 一个需要屏蔽的、按一天中时段重复出现的时间窗口，以一个时区加上起止时刻的形式给出。用它来排除你预期中的停滞时段，比如市场闭市的夜间时段，从而让那段安静的区间不会扭曲序列。

水平 vs. 增长

有些序列最好用其水平来建模；另一些则最好用其相对变化来建模。use_diffs 设置（即对数差分变换）会让模型转为基于对数增长——也就是从一步到下一步的比例变化——而非原始数值来工作。

对于呈乘性变动或在很宽范围内强烈趋势化的序列，请打开它：营收、价格，以及任何 10% 的变动比绝对变动更重要的场景。对于位于一个有界、大致平稳范围内、原始水平本身就是自然预测对象的序列，则保持关闭。

就绪状态

数据画像并不是你创建的那一刻就可用。Temporis 会先计算归一化统计量——学习你数据的典型分布范围，以便把数值缩放到模型期望的范围内。在它拥有足够数据完成这一步之前，数据画像尚未就绪，控制台会显示其就绪状态。

针对一个尚未就绪的数据画像进行预测会返回 HTTP 404 "Data profile is not ready to serve." 请等到控制台报告该数据画像已就绪，再调用 predict。
