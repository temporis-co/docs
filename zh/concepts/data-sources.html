文档/概念/数据源

数据源

数据源是存放你原始观测值的地方。本页介绍 Temporis 的数据模型，以及如何妥善地将数据发送进去。

你在控制台中创建一次数据源，随后便可在观测值产生时通过 API 将它们流式写入其中。

数据模型

Temporis 以行的形式存储数据。每个观测值都是一个三元组：某物是何时被测量的、被测量的是什么，以及你测得的那个数字。

timestamp
以秒为单位的 Unix 时间（整数）。观测发生的时刻。
name
该数值所属的序列——例如 orders 或 revenue。
value
一个数字。浮点数也可以。

单个数据源可以并排容纳许多命名序列。name 字段用于将它们区分开。你的数据与其他每位客户的数据相互隔离——一个数据源只属于你一人。

创建数据源

你在控制台的 Data Sources 下创建数据源。给它起个名字，它就可以接收数据了。

名称遵循以下几条规则：

  • 以小写字母开头。
  • 只使用小写字母、数字和下划线。
  • 最多 48 个字符。

因此 store_metrics 和 grid_load_eu 是有效的；而 Store-Metrics 和 2024data 则无效。一个刚创建的数据源在你向它发送观测值之前是空的。

发送观测值

你通过导入端点添加数据。每次调用都携带目标 data_source 和一组 records。这里我们向 store_metrics 推送几条 orders 读数。

curl -X POST https://api.temporis.co/v1/data_sources/ingest \
  -H "Authorization: Bearer ts_<token>" \
  -H "Content-Type: application/json" \
  -d '{
    "data_source": "store_metrics",
    "records": [
      { "timestamp": 1718870400, "name": "orders", "value": 42 },
      { "timestamp": 1718874000, "name": "orders", "value": 51 },
      { "timestamp": 1718877600, "name": "orders", "value": 47 }
    ]
  }'
import requests

requests.post(
    "https://api.temporis.co/v1/data_sources/ingest",
    headers={"Authorization": "Bearer ts_<token>"},
    json={
        "data_source": "store_metrics",
        "records": [
            {"timestamp": 1718870400, "name": "orders", "value": 42},
            {"timestamp": 1718874000, "name": "orders", "value": 51},
            {"timestamp": 1718877600, "name": "orders", "value": 47},
        ],
    },
)  # -> 204 No Content
await fetch("https://api.temporis.co/v1/data_sources/ingest", {
  method: "POST",
  headers: {
    "Authorization": "Bearer ts_<token>",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    data_source: "store_metrics",
    records: [
      { timestamp: 1718870400, name: "orders", value: 42 },
      { timestamp: 1718874000, name: "orders", value: 51 },
      { timestamp: 1718877600, name: "orders", value: 47 },
    ],
  }),
}); // -> 204 No Content

一次成功的导入会返回 HTTP 204 No Content，响应体为空——没有什么内容可读回，这些行只是被存储起来。完整的请求和错误详情，参见导入参考。

一个数据源中的多条序列

你并不限于每个数据源只放一条序列。向同一个 data_source 发送不同的 name 值，它们会作为各自独立的序列共存。一家门店可能在 store_metrics 中同时跟踪订单、营收和访客数。

{
  "data_source": "store_metrics",
  "records": [
    { "timestamp": 1718870400, "name": "orders",  "value": 42 },
    { "timestamp": 1718870400, "name": "revenue", "value": 1284.50 }
  ]
}

把相关的序列放在一起既方便，又便于你日后对它们进行联合预测。数据画像正是你选择实际读取其中哪些 name 的地方。

更新与回填

upsert 键是 (timestamp, name) 这一对。发送一条 timestamp 和 name 都已存在的记录会覆盖已存储的值，而不是添加一个重复项。

回填和重试是安全的

由于写入在 (timestamp, name) 上是幂等的，你可以在网络抖动后重放一个批次、纠正一个错误读数，或以任意顺序加载数月的历史——每个桶最终得到的值始终相同。

归档

当一个数据源不再收集新数据时，你可以在控制台中将它归档。归档会停止进一步的导入：针对已归档数据源的请求会返回 HTTP 409 "Data source is archived."

归档是非破坏性的。现有数据原封不动，任何基于该数据源构建的数据画像也会继续针对已存储的数据正常工作。
