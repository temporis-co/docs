预测与采样

一次预测请求会返回多个可能的未来，而不是一个。本页解释如何读取响应，以及采样控制项如何塑造它。关于确切的数据结构，参见predict 参考。

请求剖析

你针对一个数据画像进行预测，而不是针对原始的数据源。数据画像已经知道要读取哪些序列以及如何编码它们，因此一次请求只需说明使用哪个数据画像以及如何采样。四个字段就涵盖了这些：

data_profilestring必填

你某个数据画像的名称，例如 hourly_orders——或者一个公开数据画像。

countinteger必填

要抽取多少条独立的未来轨迹。至少为 1。

temperaturenumber必填

样本的分散程度。必须大于 0；大约 1.0 为中性。

top_pnumber必填

核采样截断值，取值范围为 0 到 1。越低越能裁剪掉不太可能的尾部。

下面是一个针对 hourly_orders 数据画像、请求 20 条轨迹的请求：

curl https://api.temporis.co/v1/predict \
  -H "Authorization: Bearer ts_your_token" \
  -H "Content-Type: application/json" \
  -d '{
    "data_profile": "hourly_orders",
    "count": 20,
    "temperature": 1.0,
    "top_p": 0.9
  }'
import requests

resp = requests.post(
    "https://api.temporis.co/v1/predict",
    headers={"Authorization": "Bearer ts_your_token"},
    json={
        "data_profile": "hourly_orders",
        "count": 20,
        "temperature": 1.0,
        "top_p": 0.9,
    },
)
resp.raise_for_status()
result = resp.json()
const resp = await fetch("https://api.temporis.co/v1/predict", {
  method: "POST",
  headers: {
    "Authorization": "Bearer ts_your_token",
    "Content-Type": "application/json",
  },
  body: JSON.stringify({
    data_profile: "hourly_orders",
    count: 20,
    temperature: 1.0,
    top_p: 0.9,
  }),
});
const result = await resp.json();

完整参考——每个字段、状态码和错误消息——参见predict 端点参考。

读取响应

一次成功的调用会返回 HTTP 200，包含四个顶层字段：

request_idstring

本次请求的十六进制标识符。请保留它，以备技术支持和你自己的日志使用。

data_profilestring

回显你所请求的数据画像。

predict_frominteger

预测从其开始延续的、最后一个已观测时间戳的 Unix 秒数。每条轨迹都从这个点之后开始。

predictionsarray

每条采样轨迹对应一个条目（长度等于 count）。每个条目都带有一个 loss 和一个由预测行组成的 data 数组。

下面的示例展示了一次 orders 预测中的两条轨迹。该数据画像的 interval 为 3600，patch_spacing 为 2，因此各行以 interval × patch_spacing = 7200 秒（两小时）为步长递进：

{
  "request_id": "9f3c1a7e2b4d",
  "data_profile": "hourly_orders",
  "predict_from": 1718888400,
  "predictions": [
    {
      "loss": 0.84,
      "data": [
        { "timestamp": 1718895600, "orders": 142.0 },
        { "timestamp": 1718902800, "orders": 137.5 }
      ]
    },
    {
      "loss": 1.07,
      "data": [
        { "timestamp": 1718895600, "orders": 151.2 },
        { "timestamp": 1718902800, "orders": 129.8 }
      ]
    }
  ]
}

每一行都有一个 timestamp，外加数据画像所读取的每条序列对应的一个字段——这里只有 orders。各时间戳以 interval × patch_spacing 秒为步长递进，所有轨迹的步长都相同，因此你可以逐行对齐这些样本。

采样控制项

有三个参数塑造你拿回的这组未来。它们在分辨率和多样性与成本和稳定性之间进行权衡。

参数作用典型值何时调高 / 调低
count抽取的独立轨迹数。样本越多，对分布的刻画越好。20 到 100需要更平滑的区间时调高；为降低 token 成本时调低。
temperature样本的分散程度。越高越多样、越不确定；越低则越聚集在最可能的路径附近。~1.0当预测看起来过窄时调高；当它们不切实际地发散时调低。
top_p核截断值。在采样前裁剪掉不太可能的尾部。0.9 到 1.0调低以抑制出人意料的跳变；调高至接近 1.0 以保留完整分布。
成本随 count 线性增长

每个样本都是独立生成的，并按模型使用的 token 计量，因此一个 count 为 100 的请求，成本大约是 count 为 20 的请求的五倍。请权衡你所需的分辨率与成本来选择 count。

Loss：为轨迹排序

每条预测都带有一个 loss。越低意味着模型认为该轨迹越典型——越符合它根据历史所做的预期。loss 较高的路径仍然是有效的样本；它们只是不那么居中而已。

用 loss 来挑选一条有代表性的轨迹，或在你汇总整组样本时作为权重。要抓取最典型的路径：

best = min(result["predictions"], key=lambda p: p["loss"])

多变量预测

如果一个数据画像读取多个序列名称，则每个预测行都会为每个 name 携带一个字段——这些序列是联合预测的，因此它们的值在同一行内保持一致。一个同时读取 kw 和 price 的数据画像会返回如下的行：

{ "timestamp": 1718895600, "kw": 482.0, "price": 31.7 }

公开数据画像

你可以针对 Temporis 维护的共享数据画像进行预测，无需自己做任何设置——只需把公开名称作为 data_profile 传入：

{ "data_profile": "temporis/polymarket", "count": 20, "temperature": 1.0, "top_p": 0.9 }

响应的形态完全相同；每行中的序列字段与该公开数据画像所读取的内容相匹配。
